# V1 중복 추천 이력 관리 정책

> **문서 목적**: 전주 추천 종목 중복 여부를 어떤 구조로 관리할지 후보안을 비교하고, V1 권장안을 제시한다.
> 이 문서는 2단계 스키마 설계의 기준이 된다. 권장안 외의 임의 구조를 추가하면 안 된다.

---

## 확정된 정책 원칙 (변경 불가)

| 규칙 | 내용 |
|------|------|
| 직전 주 메인 제외 | 직전 주 메인 추천 5개는 이번 주 메인 추천에서 원칙적으로 제외 |
| 예외적 연속 추천 | 필요 시 제한적으로 허용 가능 (세부 조건 미확정) |
| 2주 전 재추천 가능 | 2주 전 이전 추천 종목은 재추천 가능 |
| 3주 연속 금지 | 동일 종목 3주 연속 메인 추천은 금지 |

---

## 후보안 비교

### 후보안 A: 별도 overlap_history 파일 (`admin/overlap_history.json`)

**구조 개요**
- `admin/` 폴더에 별도 파일 1개를 운영한다.
- 최근 3주분의 에디션별 메인 추천 종목 티커 목록만 보관한다.
- draft 생성 시 이 파일을 읽어 중복 여부를 판단한다.
- 신규 에디션이 PUBLISHED(current 전환)될 때 자동으로 업데이트한다.

**장점**
- 중복 이력을 한 파일에서 빠르게 조회 가능
- manifest.json과 approval.json을 오염시키지 않음
- 파일 구조가 단순하고 사람이 읽기 쉬움
- 추후 DB 전환 시 이 파일만 테이블로 마이그레이션하면 됨
- archive를 조회하지 않아도 됨

**단점**
- 관리 파일이 1개 추가됨
- current 전환 시 이 파일도 함께 업데이트해야 하므로 동기화 실수 가능성이 있음
- 파일이 수동으로 잘못 수정될 경우 정책이 오작동할 수 있음

---

### 후보안 B: manifest.json picks[]에 플래그 포함

**구조 개요**
- draft/manifest.json의 각 pick 항목에 직전 주 연속 추천 여부 플래그를 추가한다.
- draft 생성 시 직전 current/manifest.json의 picks와 대조하여 플래그를 설정한다.
- 3주 연속 여부는 current + 직전 archive를 함께 조회하여 판단한다.

**장점**
- 별도 파일 추가 없음
- draft 파일 하나만 보면 중복 여부 확인 가능
- 개념이 직관적

**단점**
- 3주 연속 판단을 위해 archive 파일을 별도로 읽어야 함
- 플래그는 draft 생성 시점의 정보이므로, 이후 수정이 발생해도 자동 갱신이 안 됨
- manifest.json이 콘텐츠 메타 외에 운영 로직 데이터를 포함하게 됨 (역할 혼재)
- 승인 후 current로 이동하면 플래그 의미가 퇴색됨

---

### 후보안 C: archive 직접 파생 (별도 파일 없음)

**구조 개요**
- 별도 이력 파일을 만들지 않는다.
- draft 생성 또는 admin 검수 시 `data/archive/`의 최근 2-3개 에디션 manifest를 직접 읽어 중복 여부를 판단한다.
- 이력 기록 없이 archive가 그 자체로 이력의 역할을 한다.

**장점**
- 추가 파일 없음 (가장 단순한 파일 구조)
- archive가 정확한 진실의 원천(source of truth)
- 별도 동기화 필요 없음

**단점**
- 중복 판단 시 매번 archive 폴더의 여러 파일을 탐색해야 함
- archive 구조 변경 시 중복 검사 로직도 함께 변경해야 함
- admin이 수동으로 확인해야 하는 경우 여러 파일을 비교해야 하므로 실수 가능성이 높음
- 파일 기반 V1에서 archive가 쌓일수록 탐색 비용이 증가

---

### 후보안 D: approval 중심 수동 확인

**구조 개요**
- 별도 이력 구조를 만들지 않는다.
- admin이 검수 시 직접 직전 current와 archive를 확인하여 중복 여부를 판단한다.
- 예외 허용 시 approval.json의 reviewer_note에 사유를 자유 텍스트로 기록한다.

**장점**
- 파일/필드 추가 없음
- 가장 단순한 구조

**단점**
- 중복 검사가 완전히 admin 개인의 기억과 판단에 의존함
- 검수 실수 시 정책 위반을 감지할 수 없음
- 예외 승인의 구조화된 흔적이 없어 이후 감사나 이력 추적 불가
- 3주 연속 금지 같은 규칙을 자동으로 강제할 수 없음

| 항목 | 후보안 A (별도 파일) | 후보안 B (manifest 플래그) | 후보안 C (archive 파생) | 후보안 D (수동) |
|------|---------------------|--------------------------|------------------------|----------------|

| 동기화 위험 | 있음 (낮음) | 없음 | 없음 | 없음 |

## V1 권장안: 후보안 A (별도 overlap_history 파일)

### 권장안 요약

- `admin/overlap_history.json` 파일 1개를 운영한다.
- 최근 3주분(직전 3개 에디션)의 메인 추천 티커 목록만 보관한다.
- 이력은 3주치 이상 보관할 필요가 없다. (정책상 2주 전 이후 재추천 가능이므로 3주치면 충분)
- 파일은 매 current 전환 시 업데이트된다.

### 권장 이유

1. **파일 수 최소화 원칙에서 허용 가능한 범위**: 추가되는 파일이 1개뿐이며, 역할이 명확하다.
2. **manifest와 approval의 역할 분리 유지**: manifest는 콘텐츠 메타만, approval은 검수 결정만 담는 원칙을 깨지 않는다.
3. **중복 판단이 단순하고 빠름**: draft 생성 시 파일 1개만 읽으면 되며, archive를 탐색하지 않아도 된다.
4. **3주 연속 금지 규칙 지원**: 파일에 3주분 이력이 있으므로 3주 연속 여부를 바로 확인할 수 있다.
5. **추후 DB 전환 시 마이그레이션 단순**: 이 파일의 내용이 그대로 테이블 1개로 전환된다.
6. **admin 검수 보조 역할**: admin이 수동 확인이 필요한 경우에도 이 파일 하나만 열면 된다.

---

## 권장안의 파일 구조 제안 (참고용, 확정 아님)

> 아래는 2단계 스키마 설계 시 참고를 위한 구조 예시이다.
> 필드명과 구조는 2단계에서 확정한다.

- 최상위: 에디션별 배열 구조
- 각 항목: 에디션 ID + 발행일 + 메인 추천 티커 목록 5개
- 보관 범위: 최신 3개 에디션만 유지 (오래된 항목은 삭제)
- 파일 위치: `admin/overlap_history.json` (admin only, public 비노출)

## 예외 연속 추천 시 필요한 메타 정보 (제안)

예외적으로 직전 주 종목의 연속 추천을 허용하는 경우, 아래 수준의 정보가 기록되어야 한다.
어느 파일의 어느 필드에 담을지는 2단계에서 결정한다.

| 필요 정보 | 내용 |
|----------|------|
| 예외 적용 종목 식별자 | 어떤 ticker에 예외가 적용되었는지 |
| 예외 승인 시각 | admin이 예외를 인정한 시각 |
| 예외 허용 사유 | 자유 텍스트 (예: "중요 실적 발표가 이번 주에 예정됨") |

> **미확정**: 이 정보를 approval.json에 포함할지, 별도 필드를 추가할지는 2단계에서 결정한다.
> **미확정**: 예외 허용 조건의 기준 자체는 아직 정해지지 않았다.

| 예외 연속 허용 | 개별주와 동일한 원칙 적용 (세부 기준 미확정) |
