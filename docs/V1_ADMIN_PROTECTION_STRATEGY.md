# V1 Admin 보호 방식 전략

> **문서 목적**: `/admin/*` 경로를 V1 운영 환경에서 어떻게 보호할지 후보를 비교하고, V1 기준 권장 방식을 확정하기 위한 설계 문서.
> **단계**: 전략 확정 단계 — 실제 인증/미들웨어 구현 금지. 이 문서는 구현 전 판단 기준으로 사용한다.

---

## 1. 보호 대상과 보호 목적

### 보호 대상
```
/admin/*          ← Admin 검수 화면 (현재 read-only)
/admin/review     ← draft 검수 대시보드
/admin/review/report/[ticker]  ← draft 종목 상세 검토
```

### 보호 목적
| 단계 | 목적 | 위험 수준 |
|------|------|-----------|
| 현재 (read-only) | draft 데이터(미공개 에디션 정보) 외부 노출 방지 | **중간** — 투자 정보 성격의 미발행 데이터 포함 |
| 향후 (write 추가 시) | 승인 조작, 잘못된 발행 트리거 방지 | **높음** |

> **V1 현재 단계**: 실제 write 동작이 없으므로 보호 수준 요구가 상대적으로 낮다. 단, draft 내용은 미발행 투자 정보를 포함하므로 최소한의 접근 제한은 필요하다.

## 2. 보호 방식 후보 비교

### 후보 A: Vercel Password Protection (Vercel Pro/Team 기능)

Vercel 대시보드에서 배포 전체에 비밀번호를 걸어 접근 제한하는 기능.

| 항목 | 평가 |
|------|------|
| 구현 난이도 | **매우 낮음** — 코드 없음. Vercel 대시보드 설정만 |
| 운영 편의성 | 낮음 — 전체 사이트가 보호됨. public 페이지(홈, 아카이브)도 차단됨 |
| 보안 수준 | 중간 — 배포 단위 비밀번호. 세분화 불가 |
| Vercel 적합성 | 적합하나 **Pro/Team 요금제 필요** |
| 로컬 발행 전략과의 궁합 | 무관 — 로컬 발행 스크립트는 배포 보호와 독립적 |
| V1 적합성 | **부적합** — 전체 사이트 차단. 유료 기능. /admin만 보호 불가 |

### 후보 B: Next.js Middleware + 환경변수 Secret Token

`middleware.ts`에서 `/admin/*` 경로 요청 시 쿠키 또는 헤더에 특정 토큰이 있는지 확인. 토큰은 환경변수에 저장.

접근 흐름: 요청 → Middleware 검사 → 토큰 없으면 `/api/admin-login`(또는 비밀 경로) 리디렉션 → 토큰 있으면 통과.

| 항목 | 평가 |
|------|------|
| 구현 난이도 | **낮음** — middleware.ts 10~20줄. 별도 라이브러리 불필요 |
| 운영 편의성 | 중간 — 토큰을 쿠키에 저장하면 이후 세션 유지 가능 |
| 보안 수준 | **낮음~중간** — 환경변수 토큰 유출 시 보호 무력화. HTTPS에서는 실용적 수준 |
| Vercel 적합성 | **높음** — Vercel Edge Runtime에서 동작. 무료 플랜 가능 |
| 로컬 발행 전략과의 궁합 | **높음** — 로컬 발행은 이 보호와 무관하게 직접 파일 편집 |
| V1 적합성 | **적합** — 단순하고 V1 운영 환경에 현실적 |

### 후보 C: HTTP Basic Auth via Middleware

`middleware.ts`에서 `Authorization` 헤더를 확인. 없거나 틀리면 `WWW-Authenticate` 헤더와 함께 401 반환. 브라우저가 기본 로그인 다이얼로그 표시.

| 항목 | 평가 |
|------|------|
| 구현 난이도 | **낮음** — middleware.ts 내 Base64 디코딩 로직만 필요 |
| 운영 편의성 | 낮음 — 브라우저 기본 다이얼로그. UX 불편. 브라우저마다 캐시 동작 상이 |
| 보안 수준 | **낮음~중간** — HTTPS에서는 실용적이나 credentials가 매 요청에 포함됨 |
| Vercel 적합성 | **높음** — Edge Runtime 가능. 무료 플랜 |
| 로컬 발행 전략과의 궁합 | 높음 |
| V1 적합성 | **부분 적합** — 기능적으로는 가능하나 UX가 불편하여 운영 불편함 초래 |

### 후보 D: IP Allowlist (Vercel Firewall)

Vercel Firewall 기능으로 특정 IP에서만 `/admin/*` 접근 허용.

| 항목 | 평가 |
|------|------|
| 구현 난이도 | **낮음** — Vercel 대시보드 설정 |
| 운영 편의성 | **낮음** — 동적 IP 환경(카페, VPN 등)에서 차단됨. IP 변경 시마다 설정 변경 필요 |
| 보안 수준 | **높음** — IP 수준 차단이므로 애플리케이션 레이어 돌파 불가 |
| Vercel 적합성 | Pro/Enterprise 기능 |
| 로컬 발행 전략과의 궁합 | 무관 |
| V1 적합성 | **부적합** — 유료 기능 + 고정 IP 없을 경우 운영 불가 |

### 후보 E: 로컬 전용 운영 (Admin UI 미배포)

Admin 경로를 프로덕션 배포에 포함하지 않고, 로컬 개발 서버에서만 접근하는 방식.

| 항목 | 평가 |
|------|------|
| 구현 난이도 | **낮음** — 배포 설정에서 admin 제외 또는 환경 조건 분기 |
| 운영 편의성 | **낮음** — admin 확인을 위해 항상 로컬 서버 실행 필요 |
| 보안 수준 | **매우 높음** — 외부에 노출 자체가 없음 |
| Vercel 적합성 | 해당 없음 (Vercel에 배포 안 함) |
| 로컬 발행 전략과의 궁합 | **매우 높음** — 로컬 발행 스크립트와 완벽히 일치 |
| V1 적합성 | **부분 적합** — 발행 전 admin 화면 확인을 로컬에서만 할 수 있어 팀 공유가 필요 없는 1인 운영에 적합 |

## 3. V1 기준 권장 보호 방식

### **권장: 후보 B — Next.js Middleware + 환경변수 Secret Token**

**이 방식이 V1에 적합한 이유:**
- 코드 최소화 (middleware.ts 단일 파일, 별도 라이브러리 없음)
- Vercel 무료 플랜에서 동작
- /admin/* 경로만 선택적으로 보호 가능 (public 페이지 영향 없음)
- 환경변수(`ADMIN_SECRET`)로 간단하게 토큰 관리
- 로컬 발행 스크립트는 Vercel 배포 보호와 완전히 독립적

**보호 동작 흐름 (개념):**
```
브라우저 → /admin/* 요청
  → Middleware: 쿠키에 admin_token 있는가?
      있고 올바른가? → 통과
      없거나 틀린가? → /admin/login 리디렉션
  → /admin/login: SECRET 입력 폼 (정적 페이지)
      → 올바른 SECRET 입력 시 쿠키 설정 + /admin/review 리디렉션
```

**토큰 관리 원칙:**
- 환경변수 `ADMIN_SECRET`에 충분한 길이의 랜덤 값 설정
- 쿠키 만료: 세션 단위 또는 특정 기간(예: 24시간)
- HTTPS 환경에서만 운영

## 4. Read-only 단계 vs Write 단계에서의 보호 수준 차이

| 구분 | 현재 (read-only) | 향후 (write 추가 시, V1.1+) |
|------|------------------|-----------------------------|
| 위험 | 미공개 draft 데이터 노출 | 승인 조작, 잘못된 발행 |
| 필요 보호 수준 | 낮음~중간 | 높음 |
| 권장 방식 | Middleware + Secret Token | NextAuth.js 또는 동등한 세션 인증 |
| 감사 기록 | 불필요 (git history로 충분) | 접근 로그 + 액션 로그 권장 |
| 멀티유저 지원 | 불필요 | 향후 검토 필요 |

> **결론**: V1 read-only 단계에서는 후보 B(Middleware + Secret Token)가 충분하다. write 기능이 추가되는 V1.1 시점에 반드시 인증 수준을 높여야 한다.

## 5. Vercel 배포 기준 고려사항

| 항목 | 내용 |
|------|------|
| Edge Middleware 지원 | Vercel은 Next.js Middleware를 Edge Runtime으로 실행. 속도 영향 없음 |
| 환경변수 | Vercel 대시보드에서 `ADMIN_SECRET` 환경변수 설정 (Preview/Production 분리 가능) |
| 쿠키 보안 | `Secure`, `HttpOnly`, `SameSite=Strict` 속성 설정 권장 |
| 무료 플랜 적합성 | Middleware는 무료 플랜에서 동작 가능 |
| 재배포 시 토큰 유지 | 환경변수 기반이므로 재배포 후에도 유지됨 |
| Vercel Preview 환경 | Preview 배포는 별도 `ADMIN_SECRET` 또는 Vercel 팀 Authentication으로 관리 가능 |

## 6. 로컬 관리자 도구 운영과의 관계

V1 발행 전략은 **로컬에서 직접 파일을 편집하고 git commit**하는 방식이다.

- 로컬 발행 스크립트는 Vercel 배포된 admin UI를 거치지 않는다.
- 따라서 admin 보호 방식은 로컬 발행 작업에 영향을 주지 않는다.
- Admin UI(`/admin/*`)는 **발행 전 상태 확인 대시보드**로만 사용된다.
- 실제 승인 저장(approval.json 편집)은 로컬 파일 편집 + git commit으로 이루어진다.

```
[Admin 보호 대상]
  Vercel 배포 → /admin/* 조회 화면 (read-only 확인용)
  보호 필요: draft 데이터 외부 노출 방지

[로컬 발행 작업]
  로컬 환경 → 파일 편집 + 스크립트 실행 + git push
  Vercel 보호와 무관: 직접 파일시스템 접근
```

## Self-check before implementation

### 이번 문서에서 권장 보호 방식으로 제시한 것
- **후보 B (Next.js Middleware + 환경변수 Secret Token)**: /admin/* 경로를 middleware.ts에서 쿠키 토큰 확인으로 보호. 토큰은 `ADMIN_SECRET` 환경변수 기반.

### V1에서 아직 구현하지 않기로 둔 것
- NextAuth.js 등 전체 인증 라이브러리
- 멀티유저 권한 관리
- Admin 감사 로그 파일
- 세션 DB

### 구현 전에 사용자가 최종 판단해야 하는 항목
- 쿠키 만료 시간
- `/admin/login` 페이지 범위
- Vercel Preview 환경 보호 방식
- write 기능 추가 시 인증 업그레이드 시점

### signal_review / approval / 경계를 어떻게 유지했는지
- admin 보호는 **경로 수준** 보호이며, approval/signal_review 내부 데이터 경계와 무관
- 보호 방식 문서에서 approval write나 signal_review write를 다루지 않음 (별도 write 전략 문서에서 다룸)
- admin 보호 = "접근 제어". approval = "발행 결정". 두 역할을 혼동하지 않음

### 내가 임의로 구현하거나 기존 구조를 바꾸지 않은 것
- middleware.ts 코드 작성 없음
- 기존 app 디렉토리 파일 수정 없음
- 기존 JSON 파일 수정 없음
- build/dev 실행 없음

### 현재 운영 원칙 점검
| 원칙 | 점검 결과 |
|------|-----------|
| 사람은 승인 중심 | 보호 방식이 read-only 대시보드 보호에 집중. 승인 로직 간섭 없음 |
| 파일 기반 운영 | 보호 방식은 DB 없이 환경변수 토큰만 사용 |
| 자동 수집 중심 | 보호 방식이 수집 자동화에 영향 없음 |
| 뉴스는 보완 신호 | 이 문서에서 signal_review 역할 다루지 않음 (보호 전략과 무관) |
