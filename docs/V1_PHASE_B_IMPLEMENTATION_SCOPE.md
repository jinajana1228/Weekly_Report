# V1 Phase B 구현 범위 확정

> **문서 목적**: Phase B-2 최소 수집 구현에 바로 들어갈 수 있도록, 구현 범위·순서·주의사항을
> 명확히 정의한다. "무엇을 어떤 순서로 붙일 것인가"를 확정한다.
>
> 전제: V1_DATA_CONNECTION_FINALIZATION.md에서 소스별 역할과 필드 매핑이 최종 확정되었다.
> 이 문서는 그 확정안을 바탕으로 구현 실행 순서를 정의한다.
>
> 코드 구현·API 연동·스크립트 작성은 이 문서에서 하지 않는다.
> 이 문서는 구현에 들어가기 직전 팀·운영자가 합의한 구현 범위 기준 문서다.

## 1. Phase B 전체 구조에서 Phase B-2의 위치

```
Phase A (완료):
  UI 골격·admin 보호·approval/publish CLI·운영 리허설·GitHub·Vercel 배포 완료

Phase B: 실제 데이터 연결 (현재 단계)
  Phase B-1: 데이터 연결 방식 최종 확정 설계 ← 이번 단계 (완료)
  Phase B-2: 최소 수집 구현 ← 다음 단계 (이 문서의 대상)
  Phase B-3: 수집 안정화 및 추가 필드 연결

Phase C: 추천 자동화 (Phase B 완료 후)
  Phase C-1: Hard Filter 자동 적용 로직
  Phase C-2: 추천 후보 평가 로직
  Phase C-3: 뉴스 신호 자동화 연동
```

**Phase B-2의 목표**: 운영자가 수동으로 가져오던 수치 데이터를 자동 수집 스크립트로 대체하여,
draft를 생성할 때 필요한 최소 데이터 세트가 자동으로 파일에 저장되게 한다.

## 2. Phase B-2에서 바로 구현할 범위

Phase B-2에서 구현하는 것은 **데이터 수집 스크립트 및 파일 저장 구조**다.
추천 로직, UI, admin 로직은 이 단계에서 건드리지 않는다.

### 구현 대상 요약

| 구현 항목 | 소스 | 결과 파일 위치 (개념) |
|---------|------|-------------------|
| 종목별 시세 스냅샷 | KRX | data/snapshots/{week_id}/price/{ticker}.json |
| 종목별 수급 스냅샷 | KRX | data/snapshots/{week_id}/flow/{ticker}.json |
| 거래소 지정 현황 | KRX | data/snapshots/{week_id}/exchange_status/{ticker}.json |
| 지수·업종지수 주간 등락률 | KRX | data/snapshots/{week_id}/index/domestic.json |
| 재무 데이터 (분기 갱신) | DART | data/snapshots/financials/{ticker}.json |
| 공시 목록 | DART | data/snapshots/{week_id}/disclosure/{ticker}.json |
| 글로벌 지수·VIX·환율·금리 | ECOS + FRED + 보조 | data/snapshots/{week_id}/index/global.json |

> **중요**: 위 파일 경로와 JSON 키 이름은 개념적 표기다.
> 실제 파일명·경로·JSON 스키마는 구현 단계에서 확정한다.
> 현재 존재하는 data/current/, data/draft/, data/archive/ 구조는 변경하지 않는다.
> 수집 결과 파일은 별도 snapshots 영역에 저장하여 기존 구조와 충돌하지 않게 한다.

## 3. 구현 순서 — 3단계

### 1단계: KRX 수집 (최우선)

**구현 이유**: Hard Filter 전체(6종 거래소 지정현황·적자·신규 상장·동전주·저유동성·급등)
중 5개가 KRX 데이터에 의존한다. KRX 연결 없이는 추천 엔진 자체가 동작하지 않는다.

**1단계에서 구현하는 항목**:

| 항목 | 사용 목적 | 구현 우선도 |
|------|---------|----------|
| 종목별 종가 | 동전주 필터, 가격 위치 | 최우선 |
| 종목별 5거래일·20거래일 등락률 | 급등 과열 필터 | 최우선 |
| 52주 최고가·최저가 | 가격 위치 평가 | 최우선 |
| 20거래일 일평균 거래대금 | 저유동성 필터 | 최우선 |
| 거래소 지정 현황 (6종) | Hard Block 판단 | 최우선 |
| 상장일·시장 구분 | 신규 상장 필터 | 최우선 |
| KOSPI·KOSDAQ 주간 등락률 | 시장 요약 | 최우선 |
| KOSPI 업종지수 주간 등락률 (18개 업종) | 섹터 방향성 판단 근거 | 최우선 |
| 외국인 종목별 5일·20일 순매수 | 수급 신호 | 최우선 |
| 기관 종목별 5일·20일 순매수 | 수급 신호 | 최우선 |
| 외국인·기관 KOSPI 전체 주간 순매수 | 시장 수급 방향 | 우선 |
| 이동평균 20일·60일·120일 | 가격 위치 평가 | 우선 (계산으로 도출 가능) |

**1단계 접근 방식**:
- KRX 정보데이터시스템에서 파일 다운로드(후보 B) 방식으로 즉시 시작
- 병행으로 KRX 공식 오픈 API 신청 착수 (승인 후 자동화 전환)
- pykrx는 1단계 개발·테스트용 형식 확인에 한정 사용. Hard Filter 데이터는 공식 소스 확보 후 사용.

**1단계 완료 기준**:
- 유니버스 전체 종목의 시세·수급·거래소지정현황 데이터가 파일로 저장된다.
- 저장된 데이터로 Hard Filter 6개 항목 중 적어도 5개(KRX 의존)를 수동 확인 없이 판단할 수 있다.

---

### 2단계: DART 수집

**구현 이유**: Hard Filter 적자 필터(영업이익)와 감사의견, 재무 안전성(부채비율),
단기 촉매 평가(공시 목록)가 DART에 의존한다. KRX 구성이 먼저여야 유니버스 티커 목록이 확정되고
DART 호출 대상을 확정할 수 있다.

**2단계에서 구현하는 항목**:

| 항목 | 사용 목적 | 갱신 주기 |
|------|---------|---------|
| TTM 영업이익 (직전 4분기 합산) | Hard Filter 적자 필터 | 분기 1회 |
| 최근 분기·직전 분기 영업이익 | Hard Filter 연속 적자 판단 | 분기 1회 |
| 부채비율 (D/E Ratio) | 재무 안전성 평가 | 분기 1회 |
| 매출액 전년 동기 대비 성장률 | 성장성 방향 판단 | 분기 1회 |
| 감사의견 | Hard Filter 감사의견 Hard Block | 연간 |
| 공시 목록 (최근 N건) | 단기 촉매 평가 | 주 1회 |

**2단계 접근 방식**:
- DART OpenAPI 직접 사용 (API 키 발급, 무료)
- 재무 데이터: 별도 재무제표 기준. fnlttSinglAcntAll API 기준.
- TTM 계산: 직전 4개 분기 개별 영업이익 합산 계산 로직 별도 구현 필요.
- 공시 목록: 주 1회 갱신. 중요 공시 판단은 운영자 수동 검토.
- 호출 간격: 유니버스 전체를 한 번에 호출하지 않고 종목당 간격을 두고 순차 호출.

**2단계 완료 기준**:
- 유니버스 전체 종목의 재무 데이터(TTM 영업이익·부채비율)가 파일로 저장된다.
- 공시 목록이 주 1회 자동으로 갱신된다.
- Hard Filter 적자 필터와 감사의견 판단을 자동으로 수행할 수 있다.

---

### 3단계: ECOS · FRED · 글로벌 지수 수집

**구현 이유**: 시장 요약(market_summary)에 필요한 글로벌·국내 거시 지표다.
이 없이도 Hard Filter와 추천 평가는 동작하지만, 시장 요약 문구 작성을 위한 수치 근거가 없다.

**3단계에서 구현하는 항목**:

| 항목 | 소스 | 사용 목적 |
|------|------|---------|
| USD/KRW 환율 | ECOS | 시장 요약, 수출 섹터 방향성 |
| 한국은행 기준금리 | ECOS | 국내 금리 환경 판단 |
| 미국 10년 국채금리 | FRED | 글로벌 금리 환경 판단 |
| 미연준 기준금리 (FFR) | FRED | 글로벌 통화정책 환경 |
| S&P 500·NASDAQ 주간 등락률 | 보조 소스 (Yahoo Finance) | 글로벌 시장 요약 |
| VIX 수준 | 보조 소스 | 변동성 환경 판단 |

**3단계 접근 방식**:
- ECOS: 한국은행 ECOS API 키 발급 후 HTTP 호출.
- FRED: FRED API 키 발급 후 HTTP 호출. 무료, 일 500 req 한도 확인 필요.
- 보조 소스: Yahoo Finance 공개 API. 안정성 문제 시 Alpha Vantage 또는 수동 입력으로 대체.
- 3단계 항목은 주 1회 갱신으로 충분.

**3단계 완료 기준**:
- 환율·금리·글로벌 지수 데이터가 파일로 저장된다.
- 운영자가 시장 요약 문구를 작성할 때 수치 데이터를 수동으로 찾는 작업이 사라진다.

## 4. 어떤 데이터부터 붙일지 — 우선순위 근거

### 4-1. 구현 우선순위 원칙

1. **Hard Filter 먼저**: Hard Filter가 작동하지 않으면 추천 후보를 검증할 수 없다. Hard Filter 데이터를 먼저 붙이는 것이 최우선이다.
2. **추천 엔진 입력값 그 다음**: Hard Filter 통과 후보를 평가하는 수급·가격·재무 데이터가 두 번째다.
3. **시장 요약 마지막**: 시장 요약은 운영자 문구 작성이 개입되므로, 수치 근거만 자동 제공하면 된다. 우선순위가 가장 낮다.

### 4-2. 우선순위 매트릭스

| 데이터 그룹 | 추천 엔진 의존도 | 자동화 용이성 | 구현 단계 |
|-----------|-------------|------------|---------|
| 거래소 지정현황 (Hard Block 핵심) | 매우 높음 | 높음 | **1단계** |
| 시세 (동전주·급등·가격위치) | 매우 높음 | 높음 | **1단계** |
| 거래대금 (저유동성) | 매우 높음 | 높음 | **1단계** |
| 수급 (외국인·기관 순매수) | 높음 | 높음 | **1단계** |
| 지수·업종지수 | 높음 | 높음 | **1단계** |
| 재무 (영업이익·부채비율) | 높음 | 중간 (DART API 필요) | **2단계** |
| 감사의견·공시목록 | 높음 | 중간 | **2단계** |
| 환율·금리 | 중간 | 높음 | **3단계** |
| 글로벌 지수·VIX | 중간 | 높음 (보조 소스) | **3단계** |

## 6. 구현 시 반드시 주의할 점

### 6-1. 기존 파일 구조 보호

- 현재 존재하는 `data/current/`, `data/draft/`, `data/archive/`, `data/manifests/`, `data/news_signals/` 구조를 건드리지 않는다.
- 수집 결과 데이터는 별도 디렉터리(예: `data/snapshots/`)에 저장하여 기존 구조와 충돌을 방지한다.
- 실제 파일 경로와 JSON 키 이름은 구현 착수 시점에 설계하고 확정한다.

### 6-2. 거래소 지정 현황의 이중 확인 원칙

- 수집 스크립트로 자동 수집하더라도, **발행 직전 1회 추가 수동 확인**이 필수다 (V1_HARD_FILTER_POLICY.md 5절).
- 자동 수집 결과만 믿고 발행하는 구조를 만들지 않는다. 자동 수집은 "초기 검토용"이고, 발행 직전 재확인은 "최종 검증용"이다.

### 6-3. DART TTM 계산 로직

- DART API는 TTM(최근 4분기) 영업이익을 단일 값으로 제공하지 않는다.
- 직전 4개 분기(Q1~Q4)의 개별 분기 영업이익을 각각 조회하고 합산하는 별도 계산 로직이 필요하다.
- 계산 로직 구현 시 누적 기준(1Q: 1분기 누적, 2Q: 반기 누적 등) vs 개별 분기 기준 혼동에 주의해야 한다.

### 6-4. KRX 업종지수 → 서비스 13섹터 매핑

- KRX KOSPI 업종지수는 약 18개 업종으로 분류된다. 이를 서비스 13개 섹터로 변환하는 매핑 테이블이 필요하다.
- 이 매핑 테이블은 1:1 대응이 아니며, 여러 KRX 업종을 하나의 서비스 섹터로 묶거나 분리하는 판단이 개입된다.
- 매핑 테이블은 구현 착수 전에 V1_KRX_SECTOR_MAPPING.md를 기반으로 고정 작성해야 한다.
- 이 테이블이 없으면 업종지수 수집 후 favored/cautious 섹터 판단이 불가능하다.

### 6-5. 호출 한도 분산 처리

- DART: 유니버스 종목을 한 번에 호출하지 않고 순차 처리 (종목당 0.5~1초 간격 권장).
- FRED: 일 500 req 한도. Phase B-2에서 수집하는 항목 수 × 호출 빈도로 한도 소진 여부 사전 확인.
- ECOS: API 키 발급 후 요청 한도 공식 문서에서 확인 필요.

### 6-6. 파일 덮어쓰기 충돌 방지

- V1은 파일 기반 운영이다. 수집 스크립트가 실행 중 오류로 중단되면 파일이 불완전 상태로 남을 수 있다.
- 수집 완료 후 임시 파일로 먼저 저장하고, 성공 확인 후 최종 파일로 이동하는 방식을 권장한다.
- 기존 파일을 그대로 읽어야 하는 approval.json, manifest.json 등은 수집 스크립트에서 절대 수정하지 않는다.

### 6-7. 수집 스크립트의 역할 경계

- Phase B-2의 수집 스크립트 역할은 "데이터를 가져와서 파일로 저장"까지다.
- Hard Filter 판단, 추천 후보 선정, draft 작성은 Phase B-2 스크립트에 포함하지 않는다.
- 수집 스크립트가 draft 파일을 직접 수정하거나 추천 로직을 수행하면 안 된다.

## 7. Phase C 추천 자동화로 넘어가기 위한 선행조건

Phase B가 완료되고 Phase C(추천 자동화)로 이행하기 위해서는 아래 조건이 모두 충족되어야 한다.

### 7-1. 필수 선행조건 (이 없으면 Phase C 착수 불가)

| 조건 | 확인 방법 |
|------|---------|
| KRX 데이터 수집이 주 1회 안정적으로 실행된다 | 2~3주 이상 연속 수집 성공 기록 |
| DART 재무 데이터가 유니버스 전체 종목에 대해 수집된다 | 분기 1회 전체 유니버스 갱신 확인 |
| Hard Filter 6개 항목을 판단하는 데 필요한 모든 필드가 파일에 존재한다 | V1_DATA_CONNECTION_FINALIZATION.md 10-1절 최소 필드 전체 확인 |

| KRX 업종지수 → 서비스 13섹터 매핑 테이블이 고정 완료된다 | V1_KRX_SECTOR_MAPPING.md 완성 여부 확인 |
| 발행 직전 거래소 지정 재확인 체크리스트가 수집 결과와 연동된다 | 운영자 재확인 워크플로우 확인 |

### 7-2. 권장 선행조건 (없어도 Phase C 착수 가능하나 품질 저하 가능성)

| 조건 | 대안 |
|------|------|
| 공시 목록 주 1회 자동 갱신 | 운영자 수동 모니터링으로 임시 대체 가능 |
| 이동평균 3개(20·60·120일) 자동 계산 저장 | 종가 데이터로 수동 계산 가능 |
| ETF 메타 데이터 (AUM·TER·기초지수명) 자동 갱신 | 분기 수동 갱신으로 임시 대체 가능 |

### 7-3. Phase C에서 별도 설계가 필요한 항목

Phase B가 완료되더라도 아래 항목은 Phase C에서 독립적으로 설계가 필요하다:

| 항목 | Phase C 설계 내용 |
|------|----------------|
| Hard Filter 자동 적용 로직 | 수집된 필드 값과 정책(V1_HARD_FILTER_POLICY.md)을 연결하는 판단 로직 |

## 8. Self-check before Phase B-2

### 이번 문서에서 최종 확정한 데이터 연결 범위
- Phase B-2 구현 3단계(KRX → DART → ECOS·FRED·보조)와 각 단계 완료 기준 확정
- P1(바로 구현)·P2(후순위)·P3(제외) 구현 우선순위 확정
- Phase B-2 명시적 제외 목록 확정
- Phase C 이행 선행조건 확정

### 아직 최종 구현으로 넘기지 않은 범위
- 수집 스크립트 코드 작성 (구현 단계에서 착수)
- 실제 파일 경로·JSON 키 이름 (구현 착수 시 확정)
- KRX 업종 → 서비스 섹터 매핑 테이블 실제 내용 (착수 전 별도 작성)
- DART Rate Limit 실제 수치 (API 테스트 후 확인)
- 뉴스 자동화 설계 (Phase C)
- Hard Filter 자동 적용 로직 (Phase C)

### 소스별 책임이 겹치지 않도록 어떻게 정리했는지
- 1단계·2단계·3단계 각각에 소스를 1:1 대응으로 배정.
- 동일 필드를 두 소스에서 중복 수집하지 않는다는 원칙이 V1_DATA_CONNECTION_FINALIZATION.md에서 확정됨.
- 이 문서는 그 확정안의 구현 순서만 정의.

### 뉴스 신호가 보완 신호 원칙을 유지하는지
- 뉴스 자동화를 Phase C로 명시 유보. Phase B-2에서는 수동 입력 유지.
- 뉴스 신호 부재 시 수치 기반 발행이 가능한 구조(P1 수치 데이터 먼저 연결)가 먼저 완성됨.

### 운영자 개입을 최소화하도록 어떻게 정리했는지
- P1 전체(시세·수급·재무·지수·환율·금리)를 자동 수집으로 확정.
- 운영자가 Phase B-2 완료 후 해야 할 일: 섹터 방향성 판단 입력·시장 요약 문구 작성·관심 구간 설정·approval 결정.
- 수치 수집 자체는 운영자가 하지 않는다.

### 내가 임의로 구현하거나 기존 구조를 바꾸지 않은 것
- 코드 없음 / API 연동 없음 / 스크립트 없음 / 크롤러 없음
- JSON 파일 수정 없음 / 기존 UI 수정 없음
- approval/publish/admin 로직 수정 없음
- build/dev 실행 없음 / background command 없음
- 기존 data/ 디렉터리 구조 변경 없음

### 현재 운영 원칙에 위배되지 않는지 점검한 항목
- ✅ V1은 자동 수집 중심: P1 항목 전체가 3단계 구현에 포함됨
- ✅ 사람은 승인만: 수집 스크립트가 draft 파일을 수정하거나 추천을 수행하지 않도록 명시 (6-7절)
- ✅ 뉴스는 보완 신호: 뉴스 자동화 Phase C 유보 확정. 수치 먼저.
- ✅ 뉴스 부재 발행 가능: Phase B-2가 완료되면 수치 기반 발행 구조가 먼저 갖춰짐
- ✅ approval.json 게이트: 수집 스크립트가 approval.json 수정 금지 명시 (6-6절)
- ✅ 파일 기반 구조 유지: 기존 data/ 구조를 건드리지 않고 별도 snapshots 영역 사용 (3절)
