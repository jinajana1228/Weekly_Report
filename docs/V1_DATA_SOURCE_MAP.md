# V1 데이터 소스 맵

> **문서 목적**: 시장 분석, 종목 추천, ETF 추천, admin 검수에 필요한 데이터 항목 전체를 정의하고, 각 항목의 목적·사용 위치·분류·필수 여부를 매핑한다.
> 이 문서는 설계 문서이며, 구체적 데이터 소스 선정·구현은 다음 단계에서 진행한다.
> 코드 작성, API 연동, 크롤러 구현, JSON 스키마 변경은 이 문서에서 하지 않는다.

---

## 1. 데이터 분류 체계

이 서비스의 데이터는 4개 층위로 분류한다.

| 분류 | 설명 | 예시 |
|------|------|------|
| **공식 수치 데이터** | 공적 기관이 발표·공시하는 정량 데이터. 오류 시 소스에서 수정 가능. | 주가, 거래대금, 지수, 재무제표, 거래소 지정 현황 |
| **뉴스 신호 데이터** | 시장 방향성 판단에 참고하는 정성적 신호. 구조화된 방향 신호로만 활용. 기사 전문 재배포 금지. | 정책 방향 신호, 업황 변화 신호, 섹터 이슈 신호 |
| **운영 메타 데이터** | 서비스 운영 상태를 추적하는 내부 데이터. 파일 기반으로 자체 관리. | overlap_history.json, approval.json, 에디션 메타 |
| **수동 보완 데이터** | 자동 수집이 어렵거나 판단이 필요한 데이터. 운영자가 직접 입력. | 시장 요약 문구, 섹터 방향성 판단, 관심 구간 설정 |

필수 여부는 3단계로 구분한다:
- **필수**: 없으면 추천/분석/검수가 불가능한 항목
- **권장**: 있으면 판단 품질이 높아지는 항목
- **후순위**: V1에서는 없어도 운영 가능하나 V1.1 이상에서 추가 권장

**자동 수집 적합성**: 실적 발표 일정은 공시 시스템에서 수집 가능. 정책 발표 일정과 이벤트는 운영자 수동 모니터링이 현실적이다.

### 2-6. 뉴스 신호 데이터

뉴스는 수치 데이터를 보완하는 방향 신호로만 활용한다. 기사 전문을 저장하거나 재배포하지 않는다.

| 신호 유형 | 목적 | 사용 위치 | 분류 | 필수 여부 |
|---------|------|---------|------|---------|
| 섹터 레벨 업황 방향 신호 (호재/악재 방향 및 강도) | 섹터 모멘텀 수치 판단 보완 | favored/cautious_sectors 보완, 추천 내러티브 | 뉴스 신호 | 권장 |
| 정책 방향 신호 (정책 수혜 여부 판단) | 정책 촉매 방향성 판단 | 단기 촉매 평가, favored_sectors | 뉴스 신호 | 권장 |
| 개별 종목 이슈 신호 (공시 연계) | 단기 촉매 확인 또는 리스크 사전 인지 | 단기 촉매 평가, Admin 리스크 검수 | 뉴스 신호 | 권장 |
| 글로벌 섹터 이슈 신호 | 글로벌 연동 섹터 방향성 보완 | 글로벌 시장 요약, 섹터 방향성 | 뉴스 신호 | 후순위 |

> **뉴스 신호 운영 원칙**:
> - 기사 전문 저장 또는 재배포 금지
> - 방향(bullish/bearish/neutral)과 강도(high/medium/low)로만 구조화하여 저장
> - 뉴스 신호만으로 섹터를 favored로 지정하지 않는다. 반드시 수치 데이터와 교차 확인 후 판단.
> - V1에서는 운영자가 뉴스를 모니터링하고 구조화된 신호로 직접 입력한다.

## 3. Hard Filter 데이터 맵

### 3-1. 공식 지정 이슈 종목 필터

| 데이터 항목 | 목적 | 적용 필터 | 분류 | 필수 여부 |
|-----------|------|---------|------|---------|
| 관리종목 지정 여부 | 즉시 Hard Block | V1_HARD_FILTER 2-1 | 공식 수치 (거래소) | 필수 |
| 거래정지 여부 | 즉시 Hard Block | V1_HARD_FILTER 2-1 | 공식 수치 (거래소) | 필수 |
| 감사의견 (한정/부적정/의견거절 여부) | 즉시 Hard Block | V1_HARD_FILTER 2-1 | 공식 수치 (거래소/금감원) | 필수 |
| 투자경고 지정 여부 | 즉시 Hard Block | V1_HARD_FILTER 2-1 | 공식 수치 (거래소) | 필수 |
| 투자위험 지정 여부 | 즉시 Hard Block | V1_HARD_FILTER 2-1 | 공식 수치 (거래소) | 필수 |

> 이 항목들은 발행 직전에 재확인이 필요하다. 검수 후 지정 상태가 바뀔 수 있기 때문이다 (V1_HARD_FILTER_POLICY.md 5절 참조).

- 각 항목의 목적·사용 위치·분류(공식수치/뉴스신호/운영메타/수동보완)·필수 여부
- 4개 층위 분류 체계 및 3단계 필수 여부 기준

### 이번 문서에서 권장안으로 제시한 것
- 필수/권장/후순위 3단계 우선순위 구분
- 각 항목의 자동 수집 적합성 판단 방향
- 뉴스 신호 구조화 원칙 (방향/강도만 추출, 수치 교차 필수)

### 아직 최종 확정하지 않은 것
- 구체적 데이터 소스 (어떤 API, 어떤 제공처에서 가져올지)
- KRX 섹터 → 서비스 섹터 매핑 테이블 내용
- 어닝 서프라이즈 판단 기준치
- 관심 구간 설정 방식
- 뉴스 신호 구조화 포맷 세부 필드

### 구현 전에 사용자가 최종 판단해야 하는 항목
- 데이터 소스 선정 (공개 API vs 유료 데이터 vs 수동 수집)
- 자동화 우선 범위 결정
- 뉴스 신호 구조화 레벨 결정

### 내가 임의로 구현하거나 기존 구조를 바꾸지 않은 것
- 코드 없음
- API 연동 없음
- JSON 스키마 변경 없음
- 기존 정책 문서 변경 없음
- 실제 데이터 수집 없음
- 빌드/실행 없음
