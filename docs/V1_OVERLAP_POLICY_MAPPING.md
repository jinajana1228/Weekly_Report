# V1 중복 추천 정책 및 상태 파일 연동 문서

> **문서 목적**: 전주 추천 중복 정책의 확정 원칙과, 이 정책이 운영 파일 구조에 어떻게 반영될 수 있는지 정의한다.
> **중요**: 세부 예외 기준, 파일 구조 설계 후보 등은 확정 전이며, 이 문서에서는 확정된 것과 후보안을 명확히 구분한다.

---

## 확정된 중복 추천 정책 원칙

| 규칙 | 내용 | 상태 |
|------|------|------|
| 직전 주 메인 제외 | 직전 주 메인 추천 5개는 이번 주 메인 추천에서 원칙적으로 제외 | **확정** |
| 예외적 연속 추천 | 필요 시 제한적으로 허용 가능 | **큰 방향만 확정** |
| 2주 전 재추천 | 2주 전 이전 추천 종목은 재추천 가능 | **확정** |
| 3주 연속 금지 | 동일 종목 3주 연속 메인 추천은 금지 | **확정** |

## 운영 파일 구조에의 반영 방식 (후보안)

아래는 중복 추천 정책을 파일 기반 운영 구조에 반영하기 위한 설계 후보이다.
**어떤 방식을 채택할지는 아직 확정되지 않았다.** 구현 단계에서 결정한다.

---

### 후보안 A: 별도 이력 파일 방식

- `admin/overlap_history.json` 파일을 별도로 운영한다.
- 최근 N주의 에디션별 메인 추천 종목 목록을 보관한다.
- draft 생성 시 이 파일을 참조하여 중복 여부를 판단한다.
- 장점: 중복 이력을 한 곳에서 관리 가능
- 단점: 별도 파일 유지 관리 필요, 업데이트 시점 동기화 필요

---

### 후보안 B: manifest에 플래그 포함 방식

- `data/draft/manifest.json`의 각 pick 항목에 직전 주 연속 추천 여부 플래그를 포함한다.
- draft 생성 단계에서 직전 current의 picks 목록과 대조하여 자동 설정한다.
- 예외 승인이 필요한 경우 `admin/approval.json`에 관련 필드를 추가한다.
- 장점: 이력 파일 불필요, manifest만으로 확인 가능
- 단점: 3주 연속 판단을 위해 archive도 함께 조회해야 함

---

### 후보안 C: admin 수동 확인 방식

- 파일 구조에 별도 플래그/이력을 추가하지 않는다.
- admin이 검수 시 직접 직전 current(와 필요시 archive)를 확인하여 중복 여부를 판단한다.
- 장점: 구조 단순, 파일 최소화
- 단점: admin 검수 부담 증가, 실수 가능성

---

## 예외 승인 플래그 개념 (후보안)

예외적 연속 추천을 허용하는 경우, admin의 명시적 승인 기록이 필요하다.
구현 방식(어떤 파일의 어떤 필드에 기록할지)은 위 후보안 채택 후 결정한다.

예외 승인 기록에 포함되어야 할 최소 정보:
- 예외 적용 종목 식별자
- 예외를 허용한 사유 (자유 텍스트)
- admin 확인 시각

> 예외 허용 조건의 구체적 기준은 미확정이므로, 이 기록을 언제 사용할 수 있는지는 별도 정책 문서로 확정 필요.

---

## admin 검수 체크리스트 반영

위 정책 중 확정된 항목은 현재 `V1_ADMIN_FLOW.md`의 필수 검수 항목에 반영되어 있다.
- "직전 주 메인 추천 종목이 포함되어 있는지 (원칙적 제외)" 항목

파일 구조 후보안이 확정되면 해당 파일/필드를 기준으로 체크리스트가 보완된다.

---

> **미확정**: 후보안 A / B / C 중 어떤 방식을 채택할지는 구현 단계에서 결정한다.
> **미확정**: 예외 연속 추천 허용 조건 (어떤 사유가 인정되는지)

> **미확정**: 중복 추천 시 public 화면 표시 방식
