# V1 추천 운영 정책 확정 문서

> **문서 목적**: 종목 추천 로직의 운영 방식, 유니버스 구성, 중복 추천 정책, ETF 정책 등 구현 전에 확정이 필요한 운영 정책 권장안을 제시한다.
> 이 문서는 권장안이며, 각 항목의 최종 확정은 운영자의 판단이 필요하다.
> 코드 구현, 데이터 수집, 점수 계산 엔진 작성은 이 문서에서 하지 않는다.

---

## 1. V1 유니버스 구성 방식

### 1-1. 3가지 방식 비교

#### 방식 A: 자동 스캔
전체 코스피/코스닥 상장 종목(약 2,600개+)을 Hard Filter 이후 자동으로 스캔하여 후보군을 구성한다.

| 장점 | 단점 |
|------|------|
| 종목 누락 없음. 시장 전체를 대상으로 함. | V1 파일 기반 운영에서 데이터 파이프라인 구축 필요. V1에서 현실적으로 구현하기 어려움. |
| 편향 없는 기계적 선정 가능. | 전체 종목 데이터 수집/갱신 비용 높음. |
| 추후 자동화 확장에 유리. | 초기 운영에서 데이터 품질 오류 리스크 큼. |

#### 방식 B: 수동 후보군
운영자가 매주 직접 섹터별 추천 후보 목록을 작성하고, 이 목록에서만 평가를 진행한다.

| 장점 | 단점 |
|------|------|
| V1 파일 기반 운영에 즉시 적용 가능. | 운영자의 주관적 편향이 유니버스에 개입됨. |
| 데이터 파이프라인 없이도 운영 가능. | 매주 후보 목록을 처음부터 작성하는 부담이 큼. |
| 특정 이슈 종목을 빠르게 반영할 수 있음. | 유니버스가 좁아지면 추천 다양성이 떨어짐. |

#### 방식 C: 혼합형 (고정 목록 + 매주 보완)
섹터별 고정 후보 목록(코어 유니버스)을 사전에 구성하고, 매주 시장 상황에 따라 일부 종목을 추가하거나 임시로 제외하는 방식.

| 장점 | 단점 |
|------|------|
| V1 파일 기반에서 현실적으로 운영 가능. | 고정 목록 자체를 처음 구성하는 작업이 필요. |
| 유니버스의 일관성과 운영 효율을 동시에 확보. | 고정 목록이 오래되면 시장 변화를 반영하지 못할 수 있음 (정기 갱신 필요). |
| 운영자 편향을 줄이면서도 시장 이슈를 반영 가능. | 추가/제외 결정 기준이 모호하면 운영자 부담이 커짐. |
| 추후 자동화 전환 시 고정 목록이 초기 유니버스로 활용됨. | |

---

### 1-2. V1 권장안: 혼합형 (방식 C)

**권장안 내용**:
- 섹터별 코어 유니버스를 사전 구성한다. 13개 섹터 각각 10~15개 종목을 선정하여 총 130~195개 종목의 고정 후보 목록을 만든다.
- ETF는 별도로 섹터별 대표 ETF 3~5개씩 고정 목록에 포함한다.
- 매주 draft 작성 시, 고정 목록에서 Hard Filter를 적용하고 평가를 진행한다.
- 매주 최대 5개 이내 종목을 임시 추가할 수 있다. 임시 추가 종목은 그 주 draft에만 적용되며, 고정 목록 갱신은 별도 절차로 진행한다.
- 고정 목록은 분기 1회 또는 시장 구조 변화 시 점검·갱신한다.

**권장안 이유**:
- V1은 파일 기반이므로 전체 시장 자동 스캔 인프라가 없다. 혼합형은 이 제약 안에서 현실적으로 운영할 수 있다.
- 고정 목록은 매주 유니버스를 처음부터 정의하는 부담을 줄이고, 일관성 있는 평가 기반을 유지한다.
- 임시 추가 옵션은 갑작스러운 시장 이슈(신규 정책 수혜, 돌발 호재 등)를 반영할 유연성을 제공한다.

> ⚠️ **최종 확정 필요**: 섹터별 고정 목록의 구체적 종목 수(10개 vs 15개 vs 20개), 임시 추가 허용 상한(5개 vs 3개), 고정 목록 갱신 주기.

## 2. 메인 5개 선정 운영 절차

### 확정된 절차 (단계별)

```
Step 1 | 유니버스 불러오기
  → 코어 유니버스(고정 목록) + 이번 주 임시 추가 목록 합산

Step 2 | Hard Filter 적용
  → V1_HARD_FILTER_POLICY.md 기준 적용
  → Hard Block 항목: 즉시 제거
  → Soft Flag 항목: 플래그 표시 후 유지

Step 3 | 이번 주 시장 분석 결과 반영
  → favored_sectors / cautious_sectors 확인
  → cautious_sectors 종목은 별도 표시 (제외 아님, 낮은 우선순위)

Step 4 | 후보 종목별 평가
  → 5개 평가축 적용: 단기 촉매 / 가격 위치 / 수급 신호 / 시장 적합도 / 재무 안전성
  → 평가 방식은 구현 단계에서 확정 (자동/반자동/수동)

Step 5 | 중복 추천 정책 적용
  → overlap_history.json 직전 에디션 메인 추천과 대조
  → 직전 주(n-1) 메인 추천 종목 후보에서 제거 (원칙)
  → n-1과 n-2 모두 추천된 종목 강제 제거 (3주 연속 방지)

Step 6 | 섹터 분산 강제 적용
  → 상위 후보에서 섹터 중복 없이 5개 선택
  → 섹터 중복 발생 시 동일 섹터 내 순위가 높은 종목 우선 선택

Step 7 | ETF 포함 여부 판단
  → ETF 포함이 필요한 시장 상황인지 확인 (아래 ETF 섹션 참조)
  → ETF와 동일 섹터 개별주 동시 포함 금지 재확인

Step 8 | Admin 검수
  → 선정 결과 + Soft Flag 목록 + 중복 추천 예외 여부 전달
  → Admin이 최종 발행 여부 결정

Step 9 | 발행 직전 공식 지정 재확인
  → 관리/거래정지/경고/위험 지정 상태가 검수 이후 변경되지 않았는지 재확인
```

## 3. same_sector_alternatives 운영 정책

### 3-1. 보정된 권장안: 원칙 배제 + 예외 허용 + 라벨 표시

> **[보정 이유]**: 이전 권장안에서 "완화 적용(직전 주 메인 허용)"을 채택했으나, 이 방식은 alternatives가 직전 에디션 메인 종목의 보관소가 되는 부작용이 있다. 메인(5개) + alternatives(1~2개/섹터)를 합산하면 독자는 동일 종목을 연속 에디션에서 반복 접하게 된다. 이는 추천의 순환과 다양성을 유지하려는 중복 정책 취지를 무력화한다. 따라서 alternatives에도 원칙 배제를 적용하되, 좁은 예외 조건과 라벨 표시 의무를 두는 방식으로 보정한다.

**보정된 권장안 내용**:
- same_sector_alternatives에도 직전 주(n-1) 메인 추천 종목은 **원칙적으로 배제**한다.
- 단, 아래 두 조건 중 하나를 충족하는 경우 예외적으로 포함 가능하다.

| 예외 조건 | 설명 |
|----------|------|
| 가격 하락 재진입 | 직전 추천 시점 대비 가격이 10% 이상 하락하여 관심 구간에 재진입한 경우 |
| 신규 섹터 촉매 발생 | 직전 추천 이후 섹터 레벨의 새로운 촉매(정책 발표, 업황 변화 등)가 발생한 경우 |

- 예외 포함 시 해당 종목의 `one_line_reason`에 **`[연속노출]`** 라벨을 반드시 표시한다.
- n-1, n-2 두 에디션에서 메인 또는 alternatives로 이미 노출된 종목은 n에서 alternatives 포함도 **강제 제거** (예외 없음).
- Hard Filter(Hard Block)는 alternatives에도 메인과 동일하게 적용한다.

**원칙 배제로 전환한 이유**:
- "완화 적용"은 alternatives의 인적 구성이 고착될 위험이 있다. 메인이 5개, alternatives가 1~2개씩이면 한 에디션에서 독자에게 노출되는 종목은 10~15개에 달한다. 이 중 직전 주 메인 5개가 그대로 alternatives로 이동하면 독자는 사실상 동일한 목록을 반복 접하게 된다.
- `[연속노출]` 라벨을 두는 이유는 두 가지다: 독자가 "이 종목은 지난 에디션에도 등장했다"는 맥락을 인식하게 하고, admin이 alternatives 구성 시 의식적으로 검토하게 만든다.
- 섹터별 후보가 부족하다면 코어 유니버스 확대로 해결해야 한다. alternatives 배제 원칙을 완화하는 것은 근본 해결이 아니다.

> ⚠️ **최종 확정 필요**: 예외 조건 중 "10% 이상 하락"의 구체적 수치는 운영 경험 후 조정 가능. 라벨 표시 형식(`[연속노출]`)은 구현 단계에서 확정. "n-1, n-2 두 에디션 모두 노출" 기준을 메인 한정으로 볼지 alternatives 포함으로 볼지도 확정 필요.

### 3-2. same_sector_alternatives ETF 포함 여부

- alternatives에 ETF 포함 가능하다.
- 메인에 개별주가 선정된 섹터의 alternatives로 동일 섹터 ETF를 넣는 것은 허용한다. 이는 독자가 개별주 대신 ETF로 분산 접근하는 선택지를 주기 위함이다.
- 단, ETF가 alternatives에 포함될 경우 한 줄 이유(one_line_reason)에 "ETF임"과 "추종 대상"이 명확히 드러나야 한다.

## 4. 직전 주 메인 추천 예외 허용 정책

### 4-1. 권장안: 예외 허용, 1개 상한, 조건 엄격 적용

**권장안 내용**:
- 직전 주 메인 추천 종목을 이번 주에도 메인에 포함하는 예외는 **허용한다**.
- 단, 동일 에디션 내 예외 적용 가능 종목 수는 **최대 1개**다.
- 예외 적용 가능한 조건은 아래 3가지 중 최소 1가지를 충족해야 한다.

| 예외 허용 조건 | 설명 |
|-------------|------|
| 새로운 단기 촉매 발생 | 직전 추천 이후 공개된 새로운 이벤트, 공시, 정책 변화 등이 있어야 함 |
| 가격 구간 재진입 | 직전 추천 이후 가격이 하락하여 관심 구간에 재진입한 경우 |
| 시장 분위기 전환으로 해당 섹터가 재부각 | 직전 주에는 중립이었던 섹터가 이번 주 favored로 전환된 경우 |

**최대 1개 허용 이유**:
- 5개 중 2개 이상이 직전 주와 동일하면 "이번 주 새로운 아이디어가 없다"는 독자 인식이 생긴다.
- 1개 예외는 "지속 관심이 필요한 종목이 있다"는 메시지로 수용 가능하다.

**예외 적용 시 기록 의무**:
- 예외 종목, 예외 사유, admin 승인 확인 3가지를 반드시 검수 기록에 남긴다.
- 기록 없는 예외 적용은 원칙적으로 금지한다.

> ⚠️ **최종 확정 필요**: 예외 허용 조건 3가지의 구체적 판단 기준은 운영자 확인 필요. 예외 조건을 정량화할지 정성 판단으로 남길지도 결정 필요.

## 5. 3주 연속 추천 금지의 운영 적용 방식

**확정된 원칙**: n-1, n-2 두 에디션 모두에서 메인 추천된 종목은 이번 에디션(n)에서 메인 추천 불가. 예외 없음.

**운영 적용 방식**:

```
1. draft 작성 전 overlap_history.json의 최근 3개 에디션 확인
2. n-1과 n-2 에디션 메인 추천 티커 목록 교집합 추출
3. 해당 종목들은 Step 5(중복 추천 정책 적용) 단계에서 자동 제거
4. 제거된 종목이 있다면 admin 검수 화면에 표시 (제거 이유 명시)
5. Admin은 3주 연속 제거 목록을 확인하고 재확인 서명
```

**주의사항**: 3주 연속 금지는 메인 추천 기준이다. same_sector_alternatives 노출은 별도 카운트한다. (연속 2주 메인 추천 종목이 3주차에 alternatives로만 들어가는 것은 허용한다.)

## 6. ETF 우선 추천 조건 및 최대 허용 개수

### 6-1. ETF를 개별주보다 우선 추천하는 상황

아래 상황 중 하나 이상에 해당하는 경우 ETF를 메인 추천에 적극 검토한다.

| 상황 | 설명 |
|------|------|
| 시장 전반의 변동성이 높은 주 | VIX 또는 VKOSPI 급등, 외부 충격 존재 시 개별주 집중보다 분산 접근이 합리적 |
| 특정 섹터 추세는 명확하나 종목 선택 불확실성이 큰 경우 | 섹터 방향은 우호적이나 어떤 개별주가 가장 수혜를 받을지 불명확한 경우 |
| 글로벌 지수에 연동된 방향성 추천이 더 적절한 경우 | 미국 시장 호조 시 KOSPI보다 미국 지수 추종 ETF가 독자에게 더 직접적인 수단 |
| 특정 테마/섹터 ETF가 직접 시장 트렌드를 담는 경우 | 반도체 섹터 상승 국면에서 반도체 ETF가 개별주 선정 리스크 없이 섹터를 담는 수단이 될 때 |

**ETF를 우선 추천하지 않아야 할 상황**: 단순히 "안전하니까" 또는 "개별주 고르기가 귀찮으니까"라는 이유는 ETF 우선 추천의 근거가 되지 않는다.

### 6-2. 메인 5개 중 ETF 최대 허용 개수

**권장안: 최대 2개**

| 구성 | 판단 |
|------|------|
| ETF 0개 | 완전 개별주 구성. 항상 허용. |
| ETF 1개 | 일반적 구성. 추천 이유가 명확하면 허용. |
| ETF 2개 | 시장 변동성이 높은 주 등 특수 상황에서 허용. 2개 모두 추천 이유가 명확해야 함. |
| ETF 3개 이상 | 권장하지 않음. Admin이 추가 사유 검토 필요. |

**최대 2개 이유**:
- 메인 5개 중 3개 이상이 ETF라면 이 서비스의 "개별주 + ETF 혼합 추천"이라는 정체성이 희석된다.
- ETF 2개는 시장 변동성 대응 + 글로벌 노출 또는 섹터 접근이라는 두 가지 역할로 분리 가능하다.
- ETF 3개 이상은 사실상 "ETF 포트폴리오 추천"으로 서비스 성격이 바뀐다.

> ⚠️ **최종 확정 필요**: ETF 3개 이상을 admin이 허용할 수 있는 예외 조건을 사전에 정의할지 여부.

## 8. approval.json 예외 기록 필드 방향

> **[보정 이유]**: 이전 권장안에서 `exception_picks[]`를 "운영 경험이 쌓이면 필요할 수 있는 제안 필드"로 분류했다. 그러나 예외 연속 추천과 Soft Flag 포함은 V1 운영 첫 에디션부터 발생 가능한 케이스다. 기록 필드 없이 예외를 운영하면 패턴 추적과 검증이 불가능해진다. 따라서 `exception_picks[]`는 V1 구현 시 최소 포함 필드로 상향한다.

### 8-1. V1 구현 시 최소 포함 필드: `exception_picks[]`

`exception_picks[]`는 approval.json에 V1 출시 시점부터 포함하는 필드다.

**최소 데이터 구조**:

```json
"exception_picks": [
  {
    "ticker": "005930",
    "exception_type": "consecutive_main",
    "reason": "직전 추천 이후 어닝 서프라이즈 공시 발생, 관심 구간 재진입",
    "approved_by": "admin",
    "approved_at": "2026-03-24T14:30:00"
  }
]
```

| 필드 | 타입 | 설명 |
|------|------|------|
| `ticker` | string | 예외가 적용된 종목 티커 |
| `exception_type` | enum | `consecutive_main` / `soft_flag` / `etf_overcount` / `cautious_sector` |
| `reason` | string | 예외 허용 구체적 사유 (자유 텍스트, 최소 10자 이상) |
| `approved_by` | string | 검수자 식별자 (V1에서는 이름 또는 이니셜) |
| `approved_at` | ISO 8601 | 승인 일시 |

**운영 규칙**:
- 예외 적용이 없는 에디션에는 `"exception_picks": []` (빈 배열)로 명시한다.
- `exception_type`이 `consecutive_main`인 경우 `reason`에 반드시 예외 허용 조건(새 촉매 / 가격 재진입 / 섹터 전환) 중 해당 항목을 명시한다.
- 이 필드는 읽기 전용 기록 목적이다. 이 필드의 존재가 예외를 자동 승인하지 않는다.

**이 필드가 V1부터 필요한 이유**:
- 예외 연속 추천과 Soft Flag 포함은 운영 첫 에디션부터 발생 가능하다.
- 기록 없이 예외를 누적하면 운영 패턴을 추적할 수 없고, 동일 실수가 반복되어도 인지하지 못한다.
- V1 이후 운영 데이터를 분석할 때 예외 빈도와 유형이 정책 개선의 근거가 된다.

### 8-2. V1 이후 추가 고려 필드

아래 필드는 운영 경험 후 필요성이 확인될 경우 추가한다.

| 필드 | 목적 | 추가 시점 |
|------|------|---------|
| `soft_flag_decisions[]` | Soft Flag 종목별 포함/제외 결정 및 사유 | Soft Flag 운영 케이스 다수 발생 시 |
| `etf_overcount_reason` | ETF 3개 이상 허용 시 사유 | ETF 허용 개수 초과 케이스 발생 시 |
| `review_checklist_completed` | 검수 체크리스트 완료 여부 (boolean) | 검수 자동화 지원 시 |

> ⚠️ **최종 확정 필요**: `exception_type` enum 값의 최종 목록, `reason` 최소 글자 수 제약, `approved_by` 식별자 형식은 V1 스키마 확정 단계에서 결정.

## 10. Self-check after policy revision

> 이 섹션은 최초 문서 작성 이후 진행된 정책 보정 사항을 포함한다. 보정된 내용은 본문에 `[보정 이유]` 블록으로 명시되어 있다.

### 이번 정책 보정에서 변경된 것
- **same_sector_alternatives 중복 정책 보정 (3-1절)**: "완화 적용(직전 주 메인 허용)" → "원칙 배제 + 예외 허용 + `[연속노출]` 라벨 표시"로 전환. n-1, n-2 두 에디션 모두 노출된 종목은 alternatives에서도 강제 제거(예외 없음).

- 유니버스 구성 방식: 혼합형(고정 목록 + 매주 임시 추가) 권장
