# V1 뉴스 신호 검수 원칙 및 Fallback 정책

> **문서 목적**: 자동/반자동으로 생성된 뉴스 신호 결과에 대해 admin이 무엇을 확인해야 하는지, 부족한 경우 어떻게 발행을 유지하는지를 정의한다.
> 운영자가 매주 직접 뉴스를 입력하는 구조가 아니며, 자동 생성된 결과의 검수·승인과 자동화 부족 시 수치 기반 발행 유지가 이 문서의 핵심이다.
> 이 문서는 설계 문서이며, 실제 코드 구현은 다음 단계에서 진행한다.

---

## 1. admin 뉴스 신호 검수 원칙

### 1-1. admin의 역할 범위

| 역할 | 허용 여부 |
|------|---------|
| 자동 생성된 신호의 유효성 확인 | ✅ 핵심 역할 |
| 신호 발행 반영 승인 (is_approved = true) | ✅ 핵심 역할 |
| 신호 폐기 결정 (is_discarded = true) | ✅ 핵심 역할 |
| 검수 메모 기록 (validation_note) | ✅ 보완 허용 |
| 매주 뉴스를 직접 탐색하여 입력 | ❌ V1 기본 구조 아님 |
| 기사 원문 저장 또는 요약 작성 | ❌ 금지 |
| 뉴스만으로 섹터 방향성을 확정 | ❌ 수치 데이터 교차 확인 필수 |

> **핵심 원칙**: admin은 자동 생성된 뉴스 신호를 검수하고 발행 반영 여부를 결정한다. 뉴스 신호를 직접 생성·입력하는 역할이 아니다.

---

### 1-2. 검수 항목별 확인 기준

admin이 각 뉴스 신호를 검수할 때 확인해야 하는 항목:

#### A. 기사 전문 저장 금지 실무 적용

| 확인 항목 | 기준 |
|---------|------|
| `headline_summary` 길이 및 내용 | 80자 이내 요약인지 확인. 기사 본문이 통째로 저장되어 있으면 즉시 제거 |
| `why_it_matters` 내용 | 서비스 관점의 의미 서술인지 확인. 기사 인용문 또는 원문 재배포성 내용이면 삭제 |
| 기사 URL | 참조 URL은 저장 가능하나, 본문 텍스트 전체는 저장 금지 |

#### B. 신호 방향성 (direction) 검수

| 확인 항목 | 기준 |
|---------|------|
| direction과 수치 데이터 방향 일치 여부 | 해당 섹터 업종지수 등락률, 수급 방향과 bullish/bearish가 일치하는지 확인 |
| neutral로 분류된 신호 의미 | neutral은 "판단 보류"이며 발행 반영 시 추가적인 수치 근거가 필요 |
| 방향이 수치와 충돌하는 신호 | 수치 데이터를 우선한다. 뉴스 방향이 수치와 충돌하면 `validation_note`에 기록 후 발행 반영 여부 판단 |

#### C. 신호 신뢰도 (confidence) 검수

| confidence 값 | admin 처리 기준 |
|-------------|--------------|
| `high` | 검수 후 발행 반영 가능. 단, 수치 방향과 일치하는지 확인 필수 |
| `medium` | 검수 후 발행 반영 가능. 단, `validation_note`에 출처 재확인 여부 기록 권장 |
| `low` | 기본적으로 `is_discarded = true` 처리 권장. 예외적으로 반영 시 `validation_note`에 반드시 사유 기록 |

#### D. 추천 사유와의 연결 검수 (scope=STOCK)

| 확인 항목 | 기준 |
|---------|------|
| `catalyst_connected = true` 여부 | 해당 종목이 이번 주 후보군(is_pick_candidate)에 포함된 경우, 추천 사유와 실제로 연결되는지 확인 |
| 이미 소화된 재료인지 여부 | 공시 발표 후 주가가 급등하여 이미 반영된 재료라면 단기 촉매로 사용 부적합 |
| 관심 구간(interest_range)과의 연계 | 종목이 현재 관심 구간 내에 있지 않다면 단기 촉매 보강 신호로의 가치 감소 |

#### E. ETF 신호 검수 (scope=ETF)

| 확인 항목 | 기준 |
|---------|------|
| 기초지수 관련 신호인지 확인 | ETF 자체 운용 이슈가 아니라 추종 기초지수 방향성 신호인지 구분 |
| 환헤지 여부와의 연관성 | 환율 방향 신호가 환헤지/비헤지 ETF 방향성에 영향을 주는지 확인 |
| ETF 섹터 코드 정합성 | `etf_sector_code`가 해당 ETF의 실제 분류와 일치하는지 확인 |

## 2. 중복 뉴스 처리 원칙

### 2-1. 중복 판단 기준

자동 생성 시스템이 `duplicate_group_id`로 신호를 묶는 기준:

| 중복 판단 기준 | 설명 |
|-------------|------|
| 동일 이슈 동일 소스 | 같은 기사가 복수 경로로 수집된 경우 → 즉시 중복 처리 |
| 동일 이슈 복수 소스 | 같은 사건을 여러 소스가 다루는 경우 → `duplicate_group_id`로 묶어 대표 신호 1개만 발행 반영 |
| 유사 이슈 이틀 이내 | 48시간 내 동일 섹터·종목에 같은 catalyst_type으로 수집된 신호 → 중복 의심 플래그 |

### 2-2. 중복 처리 방식

1. 자동 생성 시스템이 `duplicate_group_id`로 중복 의심 신호를 묶는다.
2. admin은 같은 `duplicate_group_id`를 가진 신호 목록을 확인한다.
3. 대표 신호 1개를 `is_approved = true`로 선택하고, 나머지는 `is_discarded = true`로 처리한다.
4. 대표 신호 선택 기준:
   - `confidence`가 가장 높은 신호
   - `source_name`이 공식 기관인 신호 우선
   - 동일하면 `published_at`이 가장 빠른 신호 (최초 발표 우선)

## 3. 낮은 신뢰도 / 루머성 / 재전송 기사 처리 원칙

### 3-1. 낮은 신뢰도 (confidence = low)

| 상황 | 처리 방법 |
|------|---------|
| 단일 비공식 소스의 미확인 정보 | `is_discarded = true`. 이후 공식 소스에서 확인되면 신규 신호로 재수집 |
| 출처가 불명확한 익명 발표 | `is_discarded = true` |
| 예외적으로 low confidence 신호를 반영하는 경우 | `validation_note`에 반영 사유 필수 기록. 발행 후 이슈 발생 시 추적 가능해야 함 |

### 3-2. 루머성 / 비확인 정보

| 상황 | 처리 방법 |
|------|---------|
| 익명 관계자 발언 기반 기사 | `is_discarded = true`. 공식 발표 전까지 사용 금지 |
| "~할 것으로 알려졌다" 수준 기사 | `is_discarded = true` |
| 소셜미디어 기반 신호 | `is_discarded = true`. V1에서는 공식·준공식 소스만 사용 |

> **원칙**: 루머성 신호는 서비스 독자 보호와 서비스 신뢰 보호를 위해 발행 반영하지 않는다. V1_DATA_SOURCE_STRATEGY.md 원칙 1("공식 수치 데이터 우선")과 일치한다.

### 3-3. 재전송 기사 / 오래된 기사

| 상황 | 처리 방법 |
|------|---------|
| `published_at`이 7일 이상 지난 기사 | `relevance_window`가 `this_week`이거나 `1_2_weeks`인 경우 `is_discarded = true` |
| 과거 기사의 재전송 (동일 본문 신규 날짜) | `is_discarded = true`. 정보 기준 날짜와 전송 날짜를 구분하여 판단 |
| `relevance_window = beyond` 신호 | 발행 반영 보류. 참고 아카이브로만 저장 |

### 3-4. why_it_matters가 빈약한 경우

| 상황 | 처리 방법 |
|------|---------|
| `why_it_matters`가 자동 생성되었으나 내용이 불명확한 경우 | admin이 `validation_note`에 "보완 필요" 기록 후 `is_approved = false` 유지. 발행 반영하지 않음 |
| `why_it_matters`가 비어 있는 경우 | 자동으로 `admin_review_needed = true`로 처리되어야 함. 내용 없이 발행 반영 금지 |
| admin이 `validation_note`로 보완 기록한 경우 | 보완 내용과 함께 `is_approved = true` 결정 가능 |

## 4. 뉴스 신호 부족 시 Fallback 발행 원칙

### 4-1. 뉴스 신호 상태 분류

발행 전 뉴스 신호 충분성을 3단계로 분류한다.

| 상태 | 정의 | 발행 처리 방침 |
|------|------|------------|
| **충분** | 시장(MARKET) 1개 이상 + 추천 종목별 STOCK 또는 SECTOR 신호 3개 이상이 is_approved | 정상 발행. 뉴스 신호 시장 요약 및 추천 카드에 보강 반영. |
| **부분적** | 일부 scope의 신호가 부족하거나 is_approved 건수가 적음 | 수치 기반 발행 유지. 뉴스 신호가 있는 부분만 선택적 반영. 발행 보류 불필요. |
| **빈약** | is_approved 신호가 0개이거나, 시장 전체 신호조차 없음 | 수치 기반 단독 발행. 뉴스 신호 섹션 생략 또는 "이번 주 주요 뉴스 신호 없음" 표시 후 발행. |

---

### 4-2. 수치 기반 단독 발행 원칙 (Fallback 발행)

**뉴스 신호 없이도 발행 가능한 조건**:

1. KRX 수치 데이터 (시세, 수급, 업종지수)가 정상 수집된 상태
2. DART 재무 데이터가 최신 분기 기준으로 유효한 상태
3. ECOS/FRED 글로벌 지표가 수집된 상태
4. 수치 기반 시장 요약 (`market_summary.phase`, `market_summary.key_variable`, `market_summary.caution_point`)이 작성 가능한 상태
5. Hard Filter와 추천 평가 5개 축이 수치 데이터만으로 실행 가능한 상태

**수치 기반 단독 발행 시 처리 방식**:

| 항목 | Fallback 처리 방식 |
|------|-----------------|
| `market_summary` | 수치 기반으로만 작성. 뉴스 신호 없어도 국면·핵심변수·주의요인 서술 가능 |
| `favored_sectors` | 업종지수 주간 등락률·수급 방향 수치만으로 결정. 뉴스 신호 없어도 판단 가능 |
| `cautious_sectors` | 동일. 수치 방향성 기반으로 결정 |
| 단기 촉매 평가 | 공시 데이터(DART)만으로 평가 진행. 뉴스 신호 부재 시 공시 기반 촉매만 반영 |
| 추천 카드 | 수치 기반 추천 사유로만 구성. 뉴스 신호 연결 없이 발행 가능 |
| ETF 설명 | ETF 메타 데이터(AUM, TER, 추종지수) 기반으로 설명 작성 가능 |

---

### 4-3. admin이 직접 입력하는 것이 아닌 최소 보완 허용 범위

뉴스 신호 Fallback 상황에서 admin이 허용할 수 있는 최소 보완 범위:

| 허용 범위 | 설명 |
|---------|------|
| `validation_note` 기록 | 자동 생성된 신호의 맥락 보완 메모. 신호 자체를 새로 작성하는 것이 아님 |
| 자동 생성 신호 중 `direction` 오분류 수정 플래그 | 방향성이 잘못 분류된 경우 `validation_note`에 "direction 재확인 필요" 기록. 직접 수정은 구현 단계에서 수정 권한 설계 후 결정 |
| 발행 전 수치 기반 시장 요약 보완 | 운영자가 수치 요약을 직접 작성하는 것은 허용. 단, 이는 "뉴스 신호 입력"이 아니라 "수치 기반 서술 작성"이며 Fallback의 정상 발행 구조임 |

**허용하지 않는 보완 행위**:
- 매주 뉴스를 검색하여 신호를 직접 작성하는 행위
- 기사 원문 또는 제목을 직접 복사하여 신호 데이터를 만드는 행위
- 뉴스 신호 없이 direction, catalyst_type, confidence를 운영자가 임의로 채워 넣는 행위

---

### 4-4. 발행 보류가 필요한 상황 정의

아래 상황에서는 뉴스 신호 문제가 아닌 **수치 데이터 문제**로 발행을 보류한다. 뉴스 신호 부족 자체는 발행 보류 사유가 아니다.

| 발행 보류 사유 | 판단 기준 |
|-------------|---------|
| KRX 시세 데이터 미수집 | Hard Filter 및 가격 위치 평가 불가 → 발행 보류 |
| DART 재무 데이터 오류 | TTM 영업이익 계산 불가 → 발행 보류 |
| 거래소 지정 현황 재확인 불가 | 발행 직전 Hard Filter 재확인 불가 → 발행 보류 |
| approval.json 미완료 | admin 검수 미완료 → 발행 보류 |

> **뉴스 신호가 빈약하다는 이유만으로 발행을 보류하지 않는다.** 뉴스 신호는 보완 신호이며, 수치 기반 발행이 가능한 상태라면 Fallback 발행을 진행한다.

## 5. Fallback 남용 방지 원칙

### 5-1. Fallback이 남용되지 않기 위한 원칙

| 원칙 | 내용 |
|------|------|
| Fallback 빈도 모니터링 | 연속 3주 이상 Fallback 발행이 발생한 경우, 뉴스 자동 생성 시스템의 동작 여부를 점검한다 |
| Fallback 상태 기록 | `approval.json`에 해당 주 뉴스 신호 상태 (충분/부분적/빈약)를 기록한다. 발행 후 운영 데이터로 활용 |
| 수치 기반 발행 품질 유지 | Fallback 발행이더라도 수치 기반 시장 요약과 추천 로직은 동일 기준으로 적용한다. Fallback이 품질 저하의 구실이 되어서는 안 된다 |
| 뉴스 신호 없음 표시 투명성 | Fallback 발행 시 독자에게 "이번 주 뉴스 신호 데이터 없음"을 명시 여부는 구현 단계에서 결정. 단, 없는 신호를 있는 것처럼 표시하는 것은 금지 |

### 5-2. Fallback 허용 범위 요약표

| 뉴스 신호 상태 | 발행 방식 | admin 추가 작업 |
|-------------|---------|-------------|
| 충분 | 수치 + 뉴스 신호 보강 발행 | 신호 검수·승인 |
| 부분적 | 수치 기반 + 유효 신호만 선택 반영 | 유효 신호 검수·승인 |
| 빈약 | 수치 기반 단독 Fallback 발행 | 발행 상태 기록만 |
| 수치 데이터 오류 | 발행 보류 | 데이터 오류 확인 후 재발행 결정 |

## 6. V1에서 권장하는 최소 운영 정책 요약

| 항목 | 정책 요약 |
|------|---------|
| 뉴스 신호 생성 | 자동/반자동 생성. 운영자 직접 입력 구조 아님 |
| admin 역할 | 자동 생성된 신호 검수·승인·폐기 결정. 직접 뉴스 입력 금지 |
| 기사 저장 기준 | headline_summary(80자 이내) + why_it_matters(150자 이내)만 저장. 본문 전문 금지 |
| 중복 신호 처리 | duplicate_group_id로 묶어 대표 신호 1개만 발행 반영 |
| 낮은 신뢰도 신호 | confidence=low는 기본 폐기. 예외 반영 시 validation_note 필수 |
| 루머성 신호 | 즉시 폐기. 공식 발표 후 재수집 |
| 뉴스 부족 시 | 수치 기반 Fallback 발행. 발행 보류 사유 아님 |
| Fallback 남용 방지 | 연속 3주 이상 Fallback 시 시스템 점검. 상태 기록 유지 |
| 뉴스 vs 수치 충돌 | 수치 우선. 충돌 신호는 validation_note에 기록 |

## 7. Self-check before implementation

### 이번 문서에서 권장 뉴스 신호 구조/검수 규칙으로 제시한 것
- admin 역할 범위: 검수·승인·폐기 결정. 직접 뉴스 입력 금지
- 신뢰도별 처리 기준 (high/medium/low)
- 중복 신호 처리 방식 (duplicate_group_id 기반 대표 신호 1개 선택)
- 루머성/재전송/오래된 기사 즉시 폐기 원칙
- Fallback 발행 3단계 분류 (충분/부분적/빈약)
- 수치 기반 단독 발행 가능 조건 5가지
- Fallback 남용 방지 원칙 (연속 3주 모니터링, 상태 기록)

### 자동/반자동 생성 대상으로 본 것
- 모든 뉴스 신호 필드 (signal_id, direction, confidence, catalyst_type 등)
- duplicate_group_id 생성 및 중복 묶음
- admin_review_needed 플래그 자동 설정

### admin 검수 전용으로 본 것
- is_approved / is_discarded 결정
- validation_note 기록 (자동 생성 내용 보완 시)
- 대표 신호 선택 (duplicate_group_id 묶음 내)

### 아직 최종 확정하지 않은 필드/규칙
- Fallback 발행 시 독자 노출 방식 (UI에서 "신호 없음" 표시 여부)
- validation_note 최소 글자 수 제약
- direction 오분류 수정 권한 (구현 단계에서 설계)
- Fallback 빈도 임계값 (연속 3주 이상 권장이나 최종 수치 미확정)
- 뉴스 신호 상태 (충분/부분적/빈약) 자동 판단 기준 수치

### 구현 전에 사용자가 최종 판단해야 하는 항목
- 뉴스 신호 자동 생성 도구/방식 선택 (크롤러 구현 방식, 소스 범위)
- Fallback 발행 시 독자에게 신호 부재를 표시할지 여부
- approval.json에 뉴스 신호 상태 기록 필드 추가 여부 및 형식
- validation_note 글자 수 제약 최종 확정
- direction 오분류 수정 권한을 admin에게 줄지 여부 (구현 단계 결정)

### 내가 임의로 구현하거나 기존 구조를 바꾸지 않은 것
- 코드 없음
- 크롤러 설계 없음
- LLM 요약 구현 없음
- 기사 본문 저장 구조 없음
- 기존 JSON 파일 수정 없음
- 기존 UI 변경 없음
- 빌드/실행 없음
- approval.json 기존 구조 수정 없음

### 현재 운영 원칙(사람은 승인만, 직접 입력 지양)에 위배되지 않는지 점검한 항목

- 4-3절에서 "최소 보완 허용 범위"를 좁게 정의하고, 허용하지 않는 행위 목록을 별도 명시

- 5-1절 Fallback 남용 방지 원칙에서 "Fallback이 품질 저하의 구실이 되어서는 안 된다"고 명시
- 뉴스 신호와 수치 데이터 충돌 시 수치 우선 원칙을 2절과 6절에서 일관되게 명시
