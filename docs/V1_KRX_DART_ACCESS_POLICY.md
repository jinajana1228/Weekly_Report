# V1 KRX · DART 데이터 접근 정책

> **문서 목적**: V1 서비스 운영에서 KRX(한국거래소)와 DART(금융감독원 전자공시)로부터 데이터를 어떤 방식으로 접근·수집할지 접근 방식 후보를 정의하고, V1 기준 권장 접근 방식을 제시한다.
> 이 문서는 설계 문서이며, 실제 API 연동 코드 작성·스크립트 구현은 다음 단계에서 진행한다.
> "권장 접근 방식"은 구현 전 운영자 최종 확인이 필요한 권장안이다.

---

## 1. KRX 데이터 접근 방식 후보

### 1-1. V1에서 KRX로부터 필요한 데이터 목록

| 데이터 항목 | 용도 | 갱신 주기 |
|-----------|------|---------|
| 종목별 시세 (종가, 등락률, 52주 고/저) | 가격 위치 평가, 동전주 필터, 급등 필터 | 일간 |
| 종목별 거래대금 (20거래일 일평균) | 저유동성 필터 | 일간 |
| KOSPI / KOSDAQ 지수 시세 | 국내 시장 요약, 주간 등락률 | 일간 |
| KOSPI 업종지수 시세 | favored/cautious 섹터 판단 | 일간/주간 |
| 투자자별 매매현황 (외국인/기관 순매수) | 수급 신호 평가, 시장 수급 방향 | 일간 |
| 거래소 지정 현황 (관리/거래정지/경고/위험/단기과열/투자주의) | Hard Filter 2-1절 전체 | 일간~실시간 |
| ETF 기본 정보 (AUM, TER, 기초지수명) | ETF 메타 데이터 | 월간~분기 |
| 상장 종목 기본 정보 (상장일, 시장 구분) | 신규 상장 필터, 시장별 기준 분기 | 변경 시 |

### 1-2. 접근 방식 후보

#### 후보 A: KRX 정보데이터시스템 공식 오픈 API 신청

**개요**: KRX 정보데이터시스템(data.krx.co.kr)에서 제공하는 공식 오픈 API를 신청하여 HTTP 호출 방식으로 데이터를 수집한다.

| 항목 | 내용 |
|------|------|
| 접근 방식 | 공식 오픈 API (REST 방식 추정) |
| 신청 절차 | KRX 정보데이터시스템 회원 가입 후 API 사용 신청 |
| 비용 | 무료 (기본 제공 범위 내), 일부 데이터셋은 유료 가능성 존재 |
| 데이터 커버리지 | 시세, 수급, 거래대금, 업종지수, ETF 메타, 거래소 지정 현황 포함 가능 |
| 신뢰도 | 최상 (거래소 공식 채널) |
| 자동화 적합성 | ✅ 적합 (API 형식, 정기 호출 가능) |

**장점**:
- 공식 소스이므로 데이터 신뢰도 최상
- 단일 소스에서 시세·수급·거래소 지정 현황 대부분을 커버 가능
- 자동화 파이프라인 구성에 적합

**단점**:
- API 신청 및 승인 절차가 있어 즉시 사용이 불가능할 수 있음
- 일 호출 한도, 분당 요청 제한 등 제약 조건이 있을 수 있음 (사전 확인 필요)
- API 문서와 실제 응답 구조가 다를 수 있어 초기 파싱 설계가 필요
- 일부 데이터셋이 별도 API 엔드포인트로 분리되어 있을 수 있음

**최종 확정 전 확인 필요사항**:
- API 신청 처리 기간 및 승인 조건
- 종목별 시세, 투자자별 수급, 거래소 지정 현황을 단일 API 세트로 조회 가능한지 여부
- 일 호출 한도 (유니버스 130~195개 종목 × 필요 데이터 항목 수 기준)
- 무료 제공 범위와 유료 데이터셋 경계

---

#### 후보 B: KRX 정보데이터시스템 정형 데이터 파일 다운로드

**개요**: KRX 정보데이터시스템에서 제공하는 CSV/Excel 형태의 정형 데이터 파일을 수동 또는 자동으로 다운로드하여 파싱한다.

| 항목 | 내용 |
|------|------|
| 접근 방식 | 파일 다운로드 (CSV/Excel) |
| 신청 절차 | 회원 가입 후 데이터 다운로드 (일부 항목은 로그인 필요) |
| 비용 | 무료 (기본 제공 범위 내) |
| 데이터 커버리지 | 시세, 수급, 투자유의 지정 현황 등 포함 |
| 신뢰도 | 최상 (거래소 공식 채널) |
| 자동화 적합성 | ⚠️ 수동 다운로드는 V1 운영에 적합, 완전 자동화는 세션 인증 처리 필요 |

**장점**:
- API 신청 없이 즉시 사용 가능
- V1 파일 기반 운영 철학과 잘 맞음 (파일 → 파싱 → 사용)
- 형식이 안정적이고 파싱 규칙 설계가 상대적으로 단순
- 주 1회 수동 다운로드로도 V1 운영 가능

**단점**:
- 완전 자동화를 위해서는 웹 세션 유지 및 로그인 자동화가 필요 (V1 범위 초과 가능성)
- 파일 다운로드 방식이기 때문에 실시간·당일 기준 데이터를 빠르게 가져오기 어려울 수 있음
- 발행 직전 거래소 지정 재확인(V1_HARD_FILTER 5절)에서 수동 다운로드에 의존하면 운영 부담 발생

#### 후보 C: pykrx 라이브러리 활용 (비공식 래퍼)

**개요**: KRX 데이터를 파이썬에서 접근하기 위한 비공식 오픈소스 라이브러리(`pykrx`)를 통해 시세·수급 데이터를 수집한다.

| 항목 | 내용 |
|------|------|
| 접근 방식 | Python 라이브러리 (`pip install pykrx`) |
| 신청 절차 | 없음 (라이브러리 설치 즉시 사용) |
| 비용 | 무료 |
| 데이터 커버리지 | 시세, 거래대금, 수급, 업종지수 등 주요 항목 포함 |
| 신뢰도 | 보통 (KRX 공식 페이지에서 데이터를 파싱하는 비공식 래퍼) |
| 자동화 적합성 | ✅ 자동화 구현 용이 (Python 코드 호출) |

**장점**:
- API 신청 없이 즉시 사용 가능
- Python 환경에서 코드 몇 줄로 시세·수급 데이터 수집 가능
- 코드 예시가 풍부하여 초기 개발 속도 빠름

**단점**:
- **비공식 소스**: KRX가 공식으로 지원하지 않아, KRX 서버 구조 변경 시 라이브러리가 갑자기 동작하지 않을 수 있음
- 공식 데이터와 응답값 차이 또는 누락이 발생할 경우 공식 채널보다 확인이 어려움
- 거래소 지정 현황(투자경고, 투자위험 등)의 커버리지가 공식 API 대비 제한적일 수 있음
- V1_DATA_SOURCE_SHORTLIST.md에서 이미 "비공식 소스"를 후순위로 분류하는 원칙과 충돌 가능

> **V1 원칙과의 충돌 주의**: V1_DATA_SOURCE_STRATEGY.md 원칙 1에서 "모든 필터·평가·검수의 판단 근거는 공식 기관이 발표한 수치 데이터여야 한다"고 명시하고 있다. pykrx는 공식 소스를 비공식 방식으로 래핑한 도구이므로 Hard Filter 판단 데이터 수집에는 주의가 필요하다.

## 2. DART OpenAPI 접근 방식 후보

### 2-1. V1에서 DART로부터 필요한 데이터 목록

| 데이터 항목 | 용도 | 갱신 주기 |
|-----------|------|---------|
| 재무제표 (분기보고서, 사업보고서) — 영업이익, 부채비율, 매출 | Hard Filter 2-2절(적자), 재무 안전성 평가 | 분기 1회 |
| 최근 공시 목록 (종목별 최근 N건) | 단기 촉매 평가, 공시 이벤트 확인 | 주 1회 이상 |
| 실적 발표 예정일 (어닝 일정) | 단기 촉매 평가 (`catalyst.earnings_date`) | 분기 |

---

### 2-2. DART OpenAPI 접근 방식

**개요**: DART OpenAPI(opendart.fss.or.kr)는 금융감독원이 공식 제공하는 공시 데이터 API다. API 키 발급 후 HTTP 호출로 재무제표·공시 목록을 수집한다.

| 항목 | 내용 |
|------|------|
| 접근 방식 | REST API (HTTP GET, JSON/XML 응답) |
| 신청 절차 | DART OpenAPI 사이트 회원 가입 후 API 키 발급 (무료) |
| 비용 | 무료 |
| 일 호출 한도 | **10,000건/일** (API 키 1개 기준, DART 공식 기준) |
| 데이터 커버리지 | 재무제표(재무상태표, 손익계산서), 공시 목록, 기업 기본 정보 등 |
| 신뢰도 | 최상 (금감원 공식 채널) |
| 자동화 적합성 | ✅ 적합 |

---

### 2-3. DART 데이터별 호출 단위 설계

#### 재무 데이터 수집 방식

| 항목 | DART API 방식 | 호출 단위 |
|------|-------------|---------|
| 영업이익 (TTM 계산용) | `fnlttSinglAcntAll` — 단일 재무제표 조회 | 종목 1개 × 분기 4개 = 4건 |
| 부채비율 (D/E Ratio) | 동일 재무제표 내 포함 | 별도 호출 불필요 |
| 매출액 | 동일 재무제표 내 포함 | 별도 호출 불필요 |

> **TTM 계산 주의**: DART는 분기별 누적·별도 재무제표를 제공한다. TTM 영업이익은 직전 4개 분기(Q1~Q4)의 개별 분기 영업이익을 합산하는 별도 계산 로직이 필요하다. 단순 API 응답값으로 TTM이 제공되지 않는다.

#### 공시 목록 수집 방식

| 항목 | DART API 방식 | 호출 단위 |
|------|-------------|---------|
| 종목별 최근 공시 목록 | `list` — 공시 목록 조회 | 종목 1개 × 1건 (최근 N건 필터링) |
| 주요 공시 분류 | 공시 구분 코드로 필터 가능 | 동일 호출 내 필터링 |

---

### 2-4. 호출 한도 충분성 검토

**유니버스 규모 기준 추정**:

| 항목 | 계산 근거 | 추정 건수 |
|------|---------|---------|
| 재무 데이터 수집 (분기 1회) | 종목 200개 × 분기 4건 | **800건/분기** |
| 공시 목록 수집 (주 1회) | 종목 200개 × 1건 | **200건/주** |
| 기타 기업 기본 정보 | 유니버스 구성 시 1회성 | **200건 이내** |

**결론**: DART 일 호출 한도 10,000건 기준으로, V1 유니버스 규모(130~200개 종목)에서 재무 데이터와 공시 목록 수집은 **충분히 감당 가능**하다. 단, 한 번에 전체 유니버스를 조회하는 방식보다 분산 호출(시간 간격을 두고 순차 호출)을 권장한다.

> **주의**: DART API는 초당·분당 호출 제한이 별도로 있을 수 있다. 구현 단계에서 DART 공식 문서의 Rate Limit 규정을 반드시 확인해야 한다.

## 3. V1 기준 권장 접근 방식

### 3-1. KRX 권장 접근 방식

**권장안: 후보 B (파일 다운로드) → V1 Day 1 즉시 운영, 후보 A (공식 API) → 병행 신청 후 전환**

**단계별 접근**:

| 단계 | 시점 | 접근 방식 | 이유 |
|------|------|---------|------|
| V1 Day 1 즉시 | 지금 ~ 운영 시작 | 파일 다운로드 (후보 B) | API 신청 처리 시간을 기다리지 않고 즉시 운영 가능. V1 파일 기반 철학과 일치. |
| V1 안정화 후 전환 | API 승인 이후 | 공식 API (후보 A) | 자동화 파이프라인 구성, Hard Filter 데이터 신뢰도 강화. |

**pykrx 활용 입장**:
- V1 테스트·프로토타이핑 단계에서 데이터 형식 확인 용도로 제한적 활용 가능.
- Hard Filter 판단용 데이터 (거래소 지정 현황 등) 수집에는 사용하지 않는다.
- 실제 서비스 운영 데이터 수집은 공식 소스(후보 A 또는 B)로 전환 전제.

---

### 3-2. DART 권장 접근 방식

**권장안: DART OpenAPI 직접 사용 (API 키 발급 즉시 가능)**

| 항목 | 권장 방식 |
|------|---------|
| 재무 데이터 | `fnlttSinglAcntAll` API로 종목별 분기 재무제표 수집. 분기 1회 전체 유니버스 갱신. |
| 공시 목록 | `list` API로 종목별 최근 공시 조회. 주 1회 갱신. |
| 호출 전략 | 유니버스 전체를 한 번에 호출하지 않고, 종목당 0.5~1초 간격으로 순차 호출 (Rate Limit 회피). |
| TTM 계산 | 최근 4개 분기 재무제표 수집 후 별도 합산 로직 구현 필요. |

## 4. KRX와 DART 역할 분담

| 데이터 항목 | 담당 소스 | 근거 |
|-----------|---------|------|
| 주가 (종가, 등락률, 52주 고/저) | **KRX** | KRX 공식 시세 데이터 |
| 거래대금 (20거래일 일평균) | **KRX** | KRX 공식 거래 데이터 |
| KOSPI / KOSDAQ 지수 | **KRX** | KRX 공식 지수 데이터 |
| KOSPI 업종지수 | **KRX** | KRX 공식 업종지수 |
| 외국인/기관 수급 (종목별 + 시장 전체) | **KRX** | KRX 투자자별 매매현황 |

## 7. Self-check before implementation

### 이번 문서에서 권장안으로 제시한 접근 방식
- KRX: V1 Day 1은 파일 다운로드(후보 B) 즉시 사용, 병행으로 공식 API(후보 A) 신청 진행 후 전환
- DART: OpenAPI 직접 사용 (API 키 발급, 분기 1회 재무 갱신, 주 1회 공시 갱신)
- pykrx: 테스트/프로토타이핑 한정. Hard Filter 데이터 수집에는 사용 금지
- KRX와 DART 역할 분담: 시세·수급·거래소 지정은 KRX, 재무·공시는 DART

### 아직 최종 확정하지 않은 항목
- KRX 공식 API 신청 착수 시점 및 처리 기간
- DART 연결 vs 별도 재무제표 기준
- ETF 구성 종목 데이터 소스 (DART vs KOFIA)
- 실적 발표 예정일 구체적 수집 방식
- 거래소 지정 현황 자동 수집과 수동 재확인의 역할 분담
- pykrx 허용 범위 최종 확정

### 구현 전에 사용자가 판단해야 하는 항목
- KRX 공식 API 신청을 지금 진행할지 여부 (신청~승인 리드타임이 V1 운영 일정에 영향)
- DART 연결/별도 재무제표 기준 선택 (유니버스 종목 구성 후 결정 가능)
- ETF 구성 종목 데이터 DART API 제공 범위 직접 확인
- 발행 직전 거래소 지정 재확인을 자동화로 대체할지, 수동 재확인을 유지할지

### 내가 임의로 구현하거나 기존 구조를 바꾸지 않은 것
- 코드 없음
- API 연동 구현 없음
- 크롤러 설계 없음
- 기존 JSON 파일 수정 없음
- 기존 정책 문서 변경 없음
- 기존 UI/컴포넌트 변경 없음
- 빌드/실행 없음
- 데이터 수집 스크립트 작성 없음

### 기존 정책 문서와 충돌하지 않는지 점검한 항목
- "공식 수치 데이터 우선" 원칙: V1_DATA_SOURCE_STRATEGY.md 원칙 1과 일치 (pykrx를 Hard Filter에서 배제하는 이유로 명시)
