# V1 데이터 소스 전략

> **문서 목적**: V1 기준 데이터 수집·관리·적재 전략을 정의한다. "어떤 데이터를 어떤 방식으로 가져올 것인가"에 대한 방향을 설계하며, 구체적 구현은 다음 단계에서 진행한다.
> 코드 작성, API 연동, 크롤러 구현, JSON 스키마 변경은 이 문서에서 하지 않는다.

---

## 1. V1 데이터 수집의 핵심 원칙

### 원칙 1: 공식 수치 데이터 우선

모든 필터, 평가, 검수의 판단 근거는 공식 기관이 발표·공시한 수치 데이터여야 한다. 뉴스 기반 판단은 수치 데이터가 없는 경우에만 보완 용도로 사용한다. 수치와 뉴스가 충돌하면 수치를 우선한다.

**적용 예시**:
- 섹터를 favored로 지정할 때, 업종 지수 주간 등락률·수급 방향이 주된 근거여야 한다. 뉴스 분위기만으로 지정하지 않는다.
- Hard Filter는 공식 지정 데이터와 공시 재무 데이터에만 의존한다. 뉴스 내용으로 필터를 강화하거나 완화하지 않는다.

### 원칙 2: 뉴스는 방향 신호로만 활용

기사 전문 저장 또는 재배포는 금지한다. 뉴스는 "방향"(bullish/bearish/neutral)과 "강도"(high/medium/low)로 구조화하여 저장한다. 뉴스 신호만으로 섹터를 favored/cautious로 지정하지 않는다. 반드시 수치 데이터와 교차 확인 후 판단한다.

### 원칙 3: V1 현실성 — 최소 구조 우선

V1은 파일 기반 운영이다. 실시간 스트리밍, DB 집적, 자동 재계산이 없다. V1에서 "필수" 데이터는 주 1회 수동 또는 반자동으로 갱신해도 서비스가 가능한 수준으로 설계한다. 자동화는 가능한 항목부터 점진적으로 추가한다. "처음부터 모든 것을 자동화"하려는 설계는 V1 운영 불가로 이어진다.

### 원칙 4: 최소 데이터 세트 먼저, 추천 로직은 그 다음

추천 엔진을 구현하기 전에, 그 엔진이 참조할 데이터 필드가 안정적으로 공급되어야 한다. 데이터 연결 없이 추천 로직을 구현하면 빈 필드로 실행되는 코드가 된다. "먼저 최소 데이터 세트를 고정하고, 그 위에 추천 로직을 올린다"는 순서를 지킨다.

## 2. 공식 소스 우선 원칙 — 소스 유형 구분

V1에서 활용 가능한 데이터 소스 유형을 우선순위 순으로 분류한다.

| 순위 | 소스 유형 | 특징 | 대표 활용 범위 |
|-----|---------|------|------------|
| 1순위 | 거래소·감독기관 공식 데이터 | 가장 신뢰도 높음. 공시 시스템(DART, KRX 등) 기반 | 상장 정보, 재무제표, 공시, 거래소 지정 현황 |
| 2순위 | 금융당국·중앙은행 통계 | 공신력 높음. 정기 발표 형식 안정적 | 금리, CPI, 수출입 통계 등 거시 지표 |
| 3순위 | 글로벌 공개 데이터 소스 | 신뢰 가능한 공개 수치. 형식 표준화 수준 다양 | 해외 지수, VIX, 환율, 유가 |
| 4순위 | ETF 운용사 공시 | 신뢰도 높으나 운용사별 형식 다양 | ETF 메타 데이터 (추종지수, TER, 구성종목 등) |
| 5순위 | 뉴스/미디어 | 참고 용도. 기사 전문 저장 금지. 방향 신호만 추출 | 섹터 이슈 방향 신호, 정책 방향 신호 |

> **중요**: 어떤 구체적 API 또는 데이터 제공처를 사용할지는 이 문서에서 확정하지 않는다. 소스 유형의 우선순위만 정의한다. 구현 단계에서 운영자가 최종 선정한다.

## 3. 자동 수집이 적합한 데이터 vs 수동 관리가 적합한 데이터

### 3-1. 자동 수집 적합 데이터

자동 수집 적합의 판단 기준:
- 공개 API 또는 공식 데이터 소스가 존재한다.
- 수치가 정형화되어 있고, 파싱 규칙이 명확하다.
- 주기적으로 갱신되며 형식이 안정적이다.
- 오류 발생 시 소스에서 재확인할 수 있다.

| 데이터 그룹 | 대표 항목 | 갱신 주기 | V1 자동화 우선순위 |
|-----------|---------|---------|---------------|
| 시세 데이터 | 주가, 등락률, 52주 고/저, 거래대금 | 일간 | 최우선 |
| 지수 데이터 | KOSPI, KOSDAQ, 글로벌 지수, VIX | 일간/주간 | 최우선 |
| 투자자별 수급 | 외국인/기관/개인 순매수 | 일간 | 최우선 |
| 거래소 지정 현황 | 관리/거래정지/경고/위험/단기과열/투자주의 | 일간~실시간 | 최우선 (발행 직전 재확인 필수) |
| 재무 데이터 | TTM 영업이익, 부채비율, 매출 추이 | 분기 | 우선 |
| 환율 | USD/KRW | 일간 | 우선 |
| 금리 | 미국 10년물, 한국 3/10년물 | 일간 | 우선 |
| ETF 메타 | AUM, TER, 추종지수, 구성종목 | 월간~분기 | 우선 |
| 업종 지수 | KRX 업종별 주간 등락률 | 일간/주간 | 우선 |

---

### 3-2. 수동 관리가 적합한 데이터

수동 관리 적합의 판단 기준:
- 구조화가 어렵거나 판단이 개입되어야 한다.
- 자동 수집 시 오류·오독 위험이 높다.
- 발생 빈도가 낮아 자동화 비용이 가치를 초과한다.
- V1 파일 기반에서 수동 입력 비용이 현실적으로 관리 가능하다.

| 데이터 그룹 | 대표 항목 | 관리 방식 | 수동 관리 이유 |
|-----------|---------|---------|------------|
| 시장 요약 문구 | "현재 국면 / 핵심 변수 / 주의 요인" | 운영자 직접 작성 | 수치 해석 + 문장 판단이 개입되는 정성 서술 |
| favored/cautious 섹터 | 해당 주 섹터 방향성 목록 | 수치 확인 후 운영자 판단 입력 | 수치 + 뉴스 신호 교차 판단이 필요함 |
| 뉴스 신호 | 섹터·종목 방향 신호 요약 | 운영자 구조화 후 입력 | 기사 전문 저장 금지. 방향 판단은 사람이 해야 함 |
| 관심 구간 (interest_price_range) | 종목별 진입 참고 구간 | 운영자 설정 | 공식 공개 자료 없음. 판단 요소 포함 |
| 추천 이유 내러티브 | pick 카드 one_line_reason, 상세 리포트 내러티브 | 운영자 작성 | 자동 생성 시 서비스 품질 저하 |
| 정책 방향 신호 | 정책 수혜 여부 판단 | 운영자 모니터링 후 입력 | 정책 문서 파싱의 비정형성 |
| ETF 운용사 추가 정보 | 국가/지역·섹터 노출 비중 등 | 운용사 공시 확인 후 입력 | 운용사별 공시 형식 다양 |
| same_sector_alternatives 이유 | alternatives 각 종목 근거 | 운영자 작성 | 판단 요소 포함 |
| KRX 업종 → 서비스 섹터 매핑 테이블 | 공식 업종 지수를 서비스 섹터로 변환 | 초기 수동 작성 후 분기 점검 | 1:1 매핑 불가, 설계 판단 필요 |

## 4. V1 파일 기반 운영과 맞는 데이터 적재 전략

### 4-1. 데이터 파일 분리 원칙

V1은 DB가 없으므로 데이터를 JSON 파일 형태로 관리한다. 파일 역할별 분리 원칙:

| 파일 유형 (개념) | 내용 | 갱신 주기 | 관리 방식 |
|-------------|------|---------|---------|
| 주간 시장 스냅샷 | 해당 주 시장 분석 수치 + favored/cautious 섹터 | 주 1회 | 운영자 작성 (draft 작성 전) |
| 섹터별 코어 유니버스 | 섹터별 고정 후보 종목 목록 | 분기 1회 | 운영자 작성 후 고정 |
| 종목별 데이터 스냅샷 | Hard Filter 및 평가용 수치 (시세, 수급, 재무) | 주 1회 | 자동 수집 후 파일 저장 |
| ETF 메타 파일 | ETF 전용 메타 데이터 | 월간~분기 | 운용사 공시 확인 후 수동 갱신 |
| overlap_history.json | 최근 3개 에디션 메인 추천 이력 | 발행 후 자동 갱신 | 운영 메타 |
| approval.json | 현재 에디션 검수 상태 | 검수 진행 중 운영자 갱신 | 운영 메타 |

> 이 파일 구조는 설계 방향 예시다. 실제 파일명·경로·JSON 스키마는 구현 단계에서 확정한다.

### 4-2. 데이터 갱신 흐름 (V1 기준, 주간 사이클)

```
D-2 ~ D-1 (추천 준비):
  ① 시세/수급/거래대금 데이터 수집 (자동 또는 반자동)
  ② 재무 데이터 확인 (분기 갱신 후 최신 상태 반영)
  ③ 거래소 지정 현황 확인 (관리/경고/과열/주의 등)
  ④ 업종 지수 및 수급 데이터 정리
  ⑤ 운영자: 뉴스 신호 구조화, favored/cautious 섹터 판단
  ⑥ 운영자: 시장 요약 문구 작성

D-1 (draft 작성):
  ① Hard Filter 적용 (수집된 데이터 기반 자동 또는 반자동)
  ② 후보 종목 평가 (수치 기반 반자동 + 운영자 최종 판단)
  ③ 추천 5개 + same_sector_alternatives 선정
  ④ 관심 구간 설정, 내러티브 작성
  ⑤ overlap_history.json 대조 (중복·3주 연속 확인)

D-0 (admin 검수 및 발행):
  ① 발행 직전 공식 지정 재확인 (관리/거래정지/경고/위험)
  ② approval.json 검수 완료 업데이트
  ③ 발행 후 overlap_history.json 업데이트
```

### 4-3. 데이터 버전 관리 원칙

- 발행된 에디션의 시장 분석 수치는 archive에 함께 보존한다.
- 발행 후 소급 수정은 하지 않는다. (발행 후 오류 발견 시 다음 에디션에서 정정 표시)
- 운영자 수동 입력 데이터는 draft에 먼저 저장하고, 발행 확정 시 current로 이동한다.
- 갱신 충돌 방지: 동일 파일을 동시에 수정하지 않는다. V1에서는 운영자가 1인이라는 전제 하에 이 위험이 제한적이다.

## 5. "처음부터 다 자동화하지 않는 이유"

### 5-1. V1 파일 기반 운영의 제약

V1은 DB, 실시간 파이프라인, 스케줄러가 없다. 모든 데이터는 파일로 관리된다. 이 환경에서 전면 자동화를 시도하면:
- 파일 덮어쓰기 충돌 위험이 있다.
- 오류 발생 시 추적·복구가 어렵다.
- 자동화 코드 자체가 새로운 실패 지점이 된다. (자동화 코드 오류 → 잘못된 데이터 자동 생성)

### 5-2. 판단 품질의 문제

뉴스 신호, 섹터 방향성, 관심 구간 등 "판단이 개입되는 데이터"는 자동화가 오히려 품질을 낮춘다. V1 서비스 품질의 핵심은 "올바른 데이터를 올바른 시점에 올바르게 입력하는 운영 규율"이다. 잘못된 자동화는 잘못된 데이터를 빠르게 대량 생산하는 도구가 된다.

### 5-3. V1의 목표: 안정적 수동 운영 → 점진적 자동화

V1에서는 수동으로도 운영 가능한 최소 구조를 먼저 확립한다. 이후 운영이 안정되면 반복되고 오류 위험이 낮은 항목부터 자동화를 추가한다.

| 단계 | 시점 | 자동화 항목 |
|-----|------|-----------|
| V1 Day 1 | 즉시 | 시세/수급/거래소 지정 현황 자동 수집 |
| V1 안정화 | 1~2개월 후 | Hard Filter 자동 적용, 재무 데이터 분기 자동 갱신 |
| V1.1 | 운영 경험 후 | favored/cautious 섹터 반자동 추천 (수치 기반 알고리즘) |
| V2 이후 | 별도 검토 | 내러티브 보조 생성, 전체 파이프라인 자동화 |

## 6. 추천 로직 구현 전에 반드시 연결되어야 할 최소 데이터 세트

추천 엔진 코드를 작성하기 전에 아래 데이터 그룹이 안정적으로 공급·관리되어야 한다.

### 6-1. Hard Filter 최소 데이터 세트

| 데이터 항목 | 필요 이유 |
|-----------|---------|
| 거래소 지정 현황 (관리/거래정지/경고/위험/단기과열/주의) | 이 데이터가 없으면 Hard Block이 작동하지 않음 |
| 최근 20거래일 일평균 거래대금 | 저유동성 필터 필수 |
| 상장 시장 구분 (KOSPI/KOSDAQ) | 저유동성 기준 분기 필수 |
| 상장일 | 신규 상장 필터 필수 |
| 현재 종가 | 동전주 필터 필수 |
| TTM 영업이익 / 최근 2개 분기 영업이익 | 적자 필터 필수 |
| 최근 5거래일/20거래일 등락률 | 급등 과열 필터 필수 |

### 6-2. 추천 평가 최소 데이터 세트

| 데이터 항목 | 필요 이유 |
|-----------|---------|
| 종목 소속 섹터 (서비스 13개 섹터) | 섹터 분산 강제·시장 적합도 평가 필수 |
| favored_sectors / cautious_sectors (해당 주) | 시장 적합도 평가 필수 |
| 52주 고/저 및 이동평균 (20/60/120일) | 가격 위치 평가 필수 |
| 외국인/기관 5일/20일 순매수 | 수급 신호 평가 필수 |
| 최근 공시 요약 (이슈 여부) | 단기 촉매 평가 필수 |

| 항목 | 미확정 이유 | 확정 방법 |

| ETF 구성 종목 데이터 소스 | 운용사별 공시 형식 다양 | 구현 단계에서 운용사별 확인 |
| 뉴스 신호 구조화 방식 | 수동 입력 형식 미결 | 운영자 판단 + 구현 단계에서 형식 확정 |
| 거래소 지정 현황 조회 방식 | 자동 조회 API vs 수동 확인 미결 | 운영자 선택 |
| 업종별 수급 데이터 소스 | 종목 수급과 업종 수급 소스가 다를 수 있음 | 구현 단계에서 확인 |

## 8. Self-check before implementation

### 이번 문서에서 내가 정의한 필수 데이터 항목
- V1 최소 데이터 세트 3개 그룹: Hard Filter 최소 / 추천 평가 최소 / 시장 분석 최소

### 이번 문서에서 권장안으로 제시한 것
- 4개 수집 원칙 (공식 우선 / 뉴스 신호 구조화 / V1 현실성 / 최소 데이터 세트 우선)
- 자동 vs 수동 관리 구분 기준 및 데이터 그룹 분류
- 공식 소스 5단계 우선순위
- 데이터 적재 파일 분리 원칙 (예시 수준)
- D-2 ~ D-0 데이터 갱신 흐름 (예시)
- 단계별 자동화 확장 로드맵

### 아직 최종 확정하지 않은 것
- 구체적 API 또는 데이터 제공처 선정
- 파일명·경로·JSON 스키마 확정
- 자동 수집 스크립트 구현 방식
- 뉴스 신호 구조화 세부 포맷

### 구현 전에 사용자가 최종 판단해야 하는 항목
- 데이터 소스 선정 (무료 vs 유료 API, 자동 vs 수동 수집)
- 자동화 우선 범위 결정
- 뉴스 신호 구조화 레벨 결정 (수동 입력 형식 확정)
- ETF 메타 데이터 수집 방식 (운용사 공시 직접 확인 vs API)

### 내가 임의로 구현하거나 기존 구조를 바꾸지 않은 것
- 코드 없음
- API 연동 없음
- 크롤러 설계 없음
- JSON 스키마 변경 없음
- 기존 정책 문서 변경 없음
- 실제 데이터 수집 없음
- 빌드/실행 없음
